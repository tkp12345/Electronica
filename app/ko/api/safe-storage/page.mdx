`safeStorage` 모듈은 **Main 프로세스** 에 살아요. 문자열을 OS 의 네이티브 자격 증명 저장소(macOS Keychain, Windows DPAPI, Linux 의 secret service)에 넘기면, 디스크에 안전하게 쓸 수 있는 암호화된 바이트를 돌려주고, 나중에 다시 원래 문자열로 되돌려 줘요.

```js filename="main.js"
const { app, safeStorage } = require('electron')

app.whenReady().then(() => {
  if (safeStorage.isEncryptionAvailable()) {
    const encrypted = safeStorage.encryptString('my-secret-token')
    // `encrypted` 는 Buffer — 디스크에 저장해도 안전해요
    const plain = safeStorage.decryptString(encrypted)
  }
})
```

<EasyNote>
`safeStorage` 는 **은행의 대여 금고** 라고 생각하면 쉬워요. 쪽지를 맡기면(`encryptString`) 은행이 당신의 OS 계정만 열 수 있는 금고에 넣고 보관 영수증(암호화된 `Buffer`)을 줘요. 나중에 영수증을 다시 들고 가면(`decryptString`) 쪽지를 돌려받죠. 열쇠는 직접 관리할 필요가 없어요 — 운영체제가 대신 맡아 줘요.
</EasyNote>

## 이벤트(Events)

`safeStorage` 모듈은 **이벤트를 발생시키지 않아요**. 작고 순수하게 함수적인 모듈이에요 — 메서드를 호출하면 결과를 돌려받는 게 전부예요.

<Callout type="info">
암호화는 앱이 준비된 뒤에만 가능하므로, 모든 호출은 `app.whenReady()` 와 `safeStorage.isEncryptionAvailable()` 확인 뒤에 두세요.
</Callout>

---

## 메서드(Methods)

### 사용 가능 여부

| 메서드 | 하는 일 |
|---|---|
| `safeStorage.isEncryptionAvailable()` | `boolean` 반환 — 현재 암호화가 가능하면 `true` (아래 플랫폼 표 참고) |
| `safeStorage.isAsyncEncryptionAvailable()` | `Promise<boolean>` 반환 — **비동기** 작업에서 암호화가 가능한지 |

`isEncryptionAvailable()` 의 플랫폼별 동작:

| 플랫폼 | 다음일 때 `true` |
|---|---|
| _(macOS)_ | Keychain 을 사용할 수 있을 때 |
| _(Windows)_ | 앱이 `ready` 이벤트를 발생시킨 뒤 |
| _(Linux)_ | `ready` 이후, OS 키링의 시크릿 키를 사용할 수 있을 때 |

```js filename="availability.js"
app.whenReady().then(() => {
  if (!safeStorage.isEncryptionAvailable()) {
    // 보안 백엔드가 없음 — 거부할지 대체할지 결정
    return
  }
  // 암호화/복호화 가능
})
```

### 암호화 & 복호화 (동기)

| 메서드 | 하는 일 |
|---|---|
| `safeStorage.encryptString(plainText)` | `string` 을 암호화해 바이트 `Buffer` 반환. 암호화 불가 시 **예외 발생** |
| `safeStorage.decryptString(encrypted)` | `encryptString` 으로 만든 `Buffer` 를 받아 원래 `string` 반환 |

```js filename="sync-roundtrip.js"
const fs = require('node:fs')

// 비밀을 디스크에 저장
const encrypted = safeStorage.encryptString('refresh-token-abc123')
fs.writeFileSync('token.bin', encrypted)

// 나중에 다시 읽기
const fromDisk = fs.readFileSync('token.bin') // Buffer
const token = safeStorage.decryptString(fromDisk)
```

<Callout type="warning">
먼저 `isEncryptionAvailable()` 을 호출하세요. 백엔드가 없으면 `encryptString` 은 **예외를 던져요** — Linux 에서는 키링이 잠겨 있으면 런타임에도 발생할 수 있어요. 또한 저장한 버퍼를 다른 기기에서 항상 복호화할 수 있다고 가정하지 마세요. 암호화 키는 현재 OS 사용자에 묶여 있어서, 암호화된 데이터는 사용자나 기기 간에 **이식되지 않아요**.
</Callout>

### 암호화 & 복호화 (비동기)

| 메서드 | 하는 일 |
|---|---|
| `safeStorage.encryptStringAsync(plainText)` | `Promise<Buffer>` 반환 — `encryptString` 의 비동기 버전 |
| `safeStorage.decryptStringAsync(encrypted)` | `{ result: string, shouldReEncrypt: boolean }` 을 담은 `Promise<Object>` 반환 |

`shouldReEncrypt` 플래그는 데이터를 저장한 이후 OS 가 내부 키를 교체했을 때 `true` 가 돼요 — 값을 다시 암호화해 저장하라는 신호예요.

```js filename="async-roundtrip.js"
const encrypted = await safeStorage.encryptStringAsync('refresh-token-abc123')

const { result, shouldReEncrypt } = await safeStorage.decryptStringAsync(encrypted)

if (shouldReEncrypt) {
  // 키가 교체됨 — 새로 암호화한 복사본을 저장
  const fresh = await safeStorage.encryptStringAsync(result)
  await saveToDisk(fresh)
}
```

### Linux 설정

| 메서드 | 하는 일 |
|---|---|
| `safeStorage.setUsePlainTextEncryption(usePlainText)` _(Linux)_ | OS 비밀번호 관리자가 없을 때, 메모리 상의 비밀번호로 대칭 키를 만들도록 강제. macOS/Windows 에서는 무동작 |
| `safeStorage.getSelectedStorageBackend()` _(Linux)_ | 사용 중인 백엔드 이름을 `string` 으로 반환 (아래 표 참고) |

`getSelectedStorageBackend()` 의 가능한 값:

| 값 | 의미 |
|---|---|
| `basic_text` | 인식되지 않은 환경, 또는 `--password-store="basic"` 로 실행 — **안전하지 않음** |
| `gnome_libsecret` | GNOME / XFCE 등의 데스크톱 |
| `kwallet` / `kwallet5` / `kwallet6` | KDE 월렛 계열 |
| `unknown` | 앱이 `ready` 를 발생시키기 전에 호출됨 |

```js filename="linux-backend.js"
app.whenReady().then(() => {
  if (process.platform === 'linux') {
    const backend = safeStorage.getSelectedStorageBackend()
    if (backend === 'basic_text') {
      // 여기서는 사실상 평문 — 경고하거나 거부
    }
  }
})
```

<Callout type="warning">
Linux 에서는 키링(GNOME Keyring, KWallet 등)이 없으면 `safeStorage` 가 조용히 `basic_text` 백엔드로 폴백할 수 있는데, 이건 **실제로 안전하지 않아요**. 진짜 비밀을 맡기기 전에 `getSelectedStorageBackend()` 를 확인하세요.
</Callout>

`safeStorage` 모듈은 **속성이 없어요** — 모든 일은 위의 메서드로 처리해요.

## 쉽게 풀어 설명

`safeStorage` 는 딱 한 가지 문제를 풀어요: **"비밀이 있는데, 읽히지 않게 디스크에 저장하고 싶다."** 인증 토큰, API 키, 로컬 DB 비밀번호 같은 것들이요.

원리는 간단해요. 문자열을 주면 잠긴 바이트 상자(`Buffer`)를 돌려줘요. 그 상자는 현재 OS 계정만 열 수 있어요. 실제 암호화 키는 운영체제가 쥐고 있으니까요 — macOS 의 Keychain, Windows 의 DPAPI, Linux 의 키링이죠. 키를 직접 볼 일도, 저장할 일도, 잃어버릴 일도 없어요.

기억할 세 가지:

1. **항상 먼저 확인하세요.** 암호화 전에 `isEncryptionAvailable()` 을 호출하세요 — 특히 Linux 에서는 보안 백엔드가 보장되지 않아요.
2. **사용자별, 기기별이에요.** 한 컴퓨터에서 암호화한 `Buffer` 는 다른 컴퓨터에서 복호화할 수 없어요. 네트워크로 비밀을 보내는 용도가 아니라 로컬 저장용이에요.
3. **Linux 가 까다로워요.** 키링이 없으면 `basic_text` 로 폴백할 수 있는데, 그건 그냥 가림막 수준이에요. `getSelectedStorageBackend()` 로 무엇을 쓰고 있는지 확인하세요.

세 이름만 외우면 끝이에요: **`isEncryptionAvailable()`**(가능한가?), **`encryptString()`**(잠그기), **`decryptString()`**(풀기). 나머지는 Linux 미세 조정과 비동기 변형을 위한 거예요.

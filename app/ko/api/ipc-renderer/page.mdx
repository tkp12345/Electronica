`ipcRenderer` 모듈은 **Renderer 프로세스** 에 살아요. 웹 페이지가 Main 프로세스와 대화하게 해 주는 `EventEmitter` 예요: 단방향 메시지를 쏘거나, 질문을 던지고 답을 기다리거나, Main 프로세스가 보내오는 메시지를 가만히 듣고 있을 수 있어요.

```js filename="renderer.js"
const { ipcRenderer } = require('electron')

// Main 프로세스에 일을 시키고 결과를 기다려요
const fileContents = await ipcRenderer.invoke('read-file', '/path/to/file')
```

<Callout type="warning">
`contextIsolation` 이 켜져 있으면(보안 기본값) 렌더러는 `require('electron')` 을 **직접 할 수 없어요**. 대신 **preload 스크립트** 에서 `contextBridge` 로 채널별로 얇은 다리만 노출해야 해요. `ipcRenderer` 객체 전체를 페이지에 넘기면 절대 안 돼요.
</Callout>

## 이벤트(Events)

`ipcRenderer` 는 `EventEmitter` 지만 정해진 이벤트 이름이 따로 없어요. 대신 채널 이름은 **여러분이** 직접 정하고, 그 채널로 메시지가 도착할 때 이벤트가 발생해요. 즉, 우리가 듣는 "이벤트" 는 그저 Main 프로세스가 메시지를 보내는 채널이에요.

| 이벤트 (채널) | 발생 시점 |
|---|---|
| _여러분이 정한 임의의 채널명_ | Main 프로세스가 `webContents.send(channel, ...)` 로 그 채널에 메시지를 보냄 |
| _`invoke` 의 응답 채널_ | 자동 처리됨 — 리스너 대신 `invoke` 의 Promise 가 resolve 됨 |

모든 리스너는 첫 번째 인자로 `IpcRendererEvent` 를 받고, 그 뒤에 보낸 쪽이 넘긴 값들이 따라와요:

```js filename="listen.js"
// Main 이 보냄: win.webContents.send('update-counter', 42)
ipcRenderer.on('update-counter', (event, value) => {
  // event = IpcRendererEvent, value = 42
  document.getElementById('counter').textContent = value
})
```

`IpcRendererEvent` 객체가 담고 있는 것:

| 속성 | 무엇인가 |
|---|---|
| `event.sender` | 메시지를 받은 `ipcRenderer` 인스턴스 |
| `event.senderId` | 메시지를 보낸 `webContents.id` (`sendTo` 방식 라우팅용) |
| `event.ports` | `postMessage` 로 함께 전달된 `MessagePort` 객체들 |

<EasyNote>
`ipcRenderer` 를 **무전기** 라고 생각하세요. 버튼이 정해져 있지 않아요 — 상대방과 "채널 이름"(예: `'update-counter'`)을 약속한 뒤, 말할 때 누르고(`send`) 들을 땐 귀에 대고 있어요(`on`). 양쪽이 맞춰야 할 건 채널 이름 하나뿐이에요.
</EasyNote>

## 메서드(Methods)

### 듣기 (들어오는 메시지)

| 메서드 | 하는 일 |
|---|---|
| `ipcRenderer.on(channel, listener)` | `channel` 을 들음; 메시지가 올 때마다 `listener(event, ...args)` 실행 |
| `ipcRenderer.once(channel, listener)` | `on` 과 같지만 **다음** 메시지 한 번만 실행하고 스스로 제거됨 |
| `ipcRenderer.addListener(channel, listener)` | `on` 의 별칭 |
| `ipcRenderer.off(channel, listener)` | `channel` 에서 특정 `listener` 제거 |
| `ipcRenderer.removeListener(channel, listener)` | `off` 의 별칭 |
| `ipcRenderer.removeAllListeners([channel])` | `channel` 의 모든 리스너 제거, 생략 시 전체 채널의 리스너 제거 |

```js filename="listeners.js"
function onMenuOpen(event, fileName) {
  openFile(fileName)
}

// 듣기 시작
ipcRenderer.on('menu-open-file', onMenuOpen)

// 나중에 듣기 중단 (예: 컴포넌트 언마운트) — 반드시 정리하세요
ipcRenderer.removeListener('menu-open-file', onMenuOpen)
```

<Callout type="warning">
컴포넌트나 반복 실행되는 코드 안에서 추가한 리스너는 꼭 제거하세요. 기존 리스너를 지우지 않고 `ipcRenderer.on('channel', ...)` 를 다시 실행하면 핸들러가 **중복** 으로 쌓이고, 메시지 하나가 올 때마다 전부 실행돼요.
</Callout>

### 보내기 (나가는 메시지)

| 메서드 | 하는 일 |
|---|---|
| `ipcRenderer.send(channel, ...args)` | Main 프로세스로 단방향 **비동기** 메시지를 쏨. 답장 없음 |
| `ipcRenderer.sendSync(channel, ...args)` | **동기적으로** 보내고 Main 의 응답을 반환 — 응답이 올 때까지 렌더러를 **멈춤** |
| `ipcRenderer.invoke(channel, ...args)` | 비동기로 보내고 Main 의 `ipcMain.handle` 반환값으로 resolve 되는 `Promise` 를 받음 |
| `ipcRenderer.sendToHost(channel, ...args)` | `send` 와 같지만 Main 이 아닌 `<webview>` 의 호스트 페이지로 보냄 |

```js filename="invoke.js"
// 현대적이고 권장되는 요청/응답 패턴
async function loadSettings() {
  // Main 쪽: ipcMain.handle('get-settings', () => readSettingsFromDisk())
  const settings = await ipcRenderer.invoke('get-settings')
  applySettings(settings)
}

// 단방향 — 보내고 잊기 (답을 기대하지 않음)
ipcRenderer.send('log-event', { type: 'click', target: 'save' })
```

<Callout type="warning">
`sendSync` 는 피하세요: Main 이 응답할 때까지 **렌더러 UI 가 얼어붙어요**. 거의 모든 경우 Promise 기반 `invoke` 를 쓰는 게 맞아요.
</Callout>

### 고급 — MessagePort 전달

| 메서드 | 하는 일 |
|---|---|
| `ipcRenderer.postMessage(channel, message, [transfer])` | `message` 를 보내며, 선택적으로 `MessagePort` 객체의 소유권을 **이전** 해 직접적이고 빠른 채널을 만듦 |

```js filename="message-port.js"
// 빠르고 사적인 파이프를 위해 채널의 한쪽 끝을 Main 에 넘겨요
const { port1, port2 } = new MessageChannel()

ipcRenderer.postMessage('port-handoff', null, [port1])

// 이제 우리가 가진 포트로 직접 대화
port2.postMessage('전용 포트로 보내는 안녕')
port2.onmessage = (e) => handleReply(e.data)
```

<EasyNote>
`invoke` 하나만 외우세요: "질문을 던지고 답을 기다린다." `send` 는 "단방향으로 외치기", `on` 은 "계속 듣기", `postMessage` 는 공용 교환대 대신 나만의 직통 전화선을 깔고 싶을 때예요.
</EasyNote>

## 속성(Properties)

`ipcRenderer` 는 공개 속성이 **없어요** — 메서드만 있는 `EventEmitter` 예요. (메시지별로 필요한 세부 정보는 리스너에 전달되는 `IpcRendererEvent` 에 담겨 와요. 위의 이벤트 표를 보세요.)

## 쉽게 풀어 설명

렌더러(웹 페이지)와 Main 프로세스(Node.js, 진짜 OS 접근)는 보안을 위해 **서로 다른 세계** 에 살아요. 상대의 변수를 읽을 수 없죠. `ipcRenderer` 는 페이지가 Main 에게 거는 **전화기** 예요.

사실 배워야 할 건 딱 세 가지예요:

1. **묻고 기다리기** → `invoke('채널', ...)`. Main 만 할 수 있는 일(파일 읽기, 네이티브 대화상자 띄우기, OS 조회)을 요청하고 답을 `await` 해요. 90% 의 경우 이게 정답이에요.
2. **단방향으로 외치기** → `send('채널', ...)`. 답이 필요 없는 일을 Main 에 알려요("사용자가 저장을 눌렀다"). Main 은 `ipcMain.on` 으로 들어요.
3. **귀를 열어 두기** → `on('채널', 핸들러)`. Main 이 뭔가를 밀어 보내면(메뉴 클릭, 진행률 업데이트) 핸들러가 반응해요. 끝나면 `removeListener` 로 정리해서 핸들러가 쌓이지 않게 하세요.

"채널" 은 그저 `'get-settings'` 같은 약속된 이름이에요 — 양쪽이 똑같이 적어야 해요. 그리고 `contextIsolation` 때문에 실제 앱에서는 이걸 페이지에서 직접 부르지 않아요: **preload 스크립트** 가 작고 이름 붙은 API(`window.api.getSettings()`)를 노출하고, 그게 대신 `ipcRenderer` 를 호출해요. 그래야 페이지를 잠가 둔 채로도 Main 에게 도움을 청할 수 있어요.

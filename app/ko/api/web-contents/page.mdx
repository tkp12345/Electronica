`webContents` 모듈은 **Main 프로세스** 에 살지만, 제어 대상은 _렌더러_ 예요 — `BrowserWindow`, `BrowserView`, `<webview>` 안에 그려지는 살아 있는 웹 페이지죠. 모듈 자체는 작은 공장이에요. **`WebContents` 인스턴스** 를 돌려주고, 그 인스턴스 하나하나가 페이지 한 개를 위한 완전한 리모컨이에요.

```js filename="main.js"
const { BrowserWindow, webContents } = require('electron')

const win = new BrowserWindow()
const contents = win.webContents // 페이지의 리모컨

contents.loadURL('https://electronjs.org')
contents.on('did-finish-load', () => {
  // 페이지 로딩이 끝났어요 — 이제 페이지와 대화해도 안전해요
})
```

<EasyNote>
`BrowserWindow` 는 **액자** 이고, `WebContents` 는 그 안의 **사진** 이에요. 창은 액자·제목 표시줄·닫기 버튼을 주지만, _페이지 자체에 관한 것_(URL, 스크립트, 스크롤 위치)은 모두 `webContents` 에 들어 있어요.
</EasyNote>

## 모듈 메서드(Module methods)

`webContents` 는 `new` 로 만들지 **않아요**. 항상 창에서, 아래 공장 메서드에서, 또는 `app` 의 `web-contents-created` 이벤트에서 인스턴스를 받아요.

| 메서드 | 돌려주는 값 |
|---|---|
| `webContents.getAllWebContents()` | 살아 있는 모든 `WebContents` 배열 (모든 창·뷰·webview·devtools) |
| `webContents.getFocusedWebContents()` | 현재 포커스를 가진 `WebContents`, 없으면 `null` |
| `webContents.fromId(id)` | 해당 `id` 의 `WebContents`, 없으면 `undefined` |
| `webContents.fromFrame(frame)` | 주어진 `WebFrameMain` 을 가진 `WebContents`, 없으면 `undefined` |
| `webContents.fromDevToolsTargetId(targetId)` | DevTools 프로토콜 target id 의 `WebContents`, 없으면 `undefined` |

```js filename="find-contents.js"
// 앱 전체의 모든 페이지를 한 번에 음소거
for (const contents of webContents.getAllWebContents()) {
  contents.setAudioMuted(true)
}
```

## 인스턴스 이벤트(Instance Events)

`contents.on('이벤트명', 핸들러)` 로 듣습니다. _(macOS)_ / _(Windows)_ / _(Linux)_ 표시는 해당 플랫폼에서만 발생해요.

### 로딩 & 이동

| 이벤트 | 발생 시점 |
|---|---|
| `did-start-loading` | 탭 스피너가 돌기 시작함 |
| `did-stop-loading` | 탭 스피너가 멈춤 |
| `dom-ready` | 최상위 프레임의 문서가 로드됨 |
| `did-finish-load` | 이동이 끝나고 페이지 `onload` 까지 발생함 |
| `did-fail-load` | 로드 실패 (`errorCode`, `errorDescription`, URL 전달) |
| `did-fail-provisional-load` | 로드가 취소됨 (예: `window.stop()`) |
| `did-frame-finish-load` | 프레임 로드 완료 |
| `page-title-updated` | 이동 중 페이지 제목이 바뀜 |
| `page-favicon-updated` | 새 파비콘 URL 을 받음 |
| `did-create-window` | `window.open()` 으로 새 창이 생성됨 |
| `will-navigate` | **메인 프레임** 이동 직전 — `preventDefault()` 로 차단 가능 |
| `will-frame-navigate` | **모든** 프레임에서 이동이 시작되려 함 |
| `did-start-navigation` | 어떤 프레임이 이동을 시작함 |
| `will-redirect` | 서버 측 리다이렉트 직전 |
| `did-redirect-navigation` | 서버 측 리다이렉트 완료 |
| `did-navigate` | 메인 프레임 이동 완료 |
| `did-frame-navigate` | 어떤 프레임 이동 완료 |
| `did-navigate-in-page` | 페이지 내 이동 발생 (앵커/해시 변경) |
| `will-prevent-unload` | `beforeunload` 핸들러가 언로드를 취소하려 함 |

```js filename="navigation.js"
contents.on('did-finish-load', () => {
  // 페이지가 완전히 로드되어 상호작용 가능해요
})

contents.on('did-fail-load', (event, errorCode, errorDescription, url) => {
  // 빈 창 대신 오프라인/오류 페이지를 보여줘요
})

// 보안: 신뢰할 수 없는 사이트로의 이동을 막아요
contents.on('will-navigate', (event, url) => {
  if (!url.startsWith('https://my-trusted-app.com')) event.preventDefault()
})
```

<Callout type="warning">
"페이지 준비 완료" 신호는 `did-finish-load` 가 맞아요 — `dom-ready` 가 **아니에요**. `dom-ready` 는 DOM 만 생겼을 뿐 이미지/스타일은 아직 로딩 중일 수 있어요. 완전히 렌더된 페이지에 의존하는 `executeJavaScript` 는 `did-finish-load` 를 기다린 뒤 호출하세요.
</Callout>

### 프로세스 & 응답성

| 이벤트 | 발생 시점 |
|---|---|
| `render-process-gone` | 렌더러 프로세스가 사라짐 (`details.reason` 전달) |
| `unresponsive` | 페이지가 멈춰 응답하지 않음 |
| `responsive` | 멈췄던 페이지가 다시 응답함 |
| `destroyed` | 이 `WebContents` 가 파괴됨 |
| `preload-error` | 프리로드 스크립트가 처리되지 않은 예외를 던짐 |

```js filename="crash-recovery.js"
contents.on('render-process-gone', (event, details) => {
  // details.reason: 'crashed' | 'killed' | 'oom' | 'launch-failed' | ...
  if (details.reason !== 'clean-exit') contents.reload()
})

contents.on('unresponsive', () => {
  // 사용자에게 기다릴지/강제 새로고침할지 제안해요
})
```

| 이벤트 | 발생 시점 |
|---|---|
| `input-event` | 입력 이벤트가 페이지로 전달됨 |

| `leave-html-full-screen` | HTML 전체화면에서 나옴 |
| `zoom-changed` | 사용자가 마우스 휠로 확대/축소를 요청함 |
| `blur` | `WebContents` 가 포커스를 잃음 |
| `focus` | `WebContents` 가 포커스를 얻음 |
| `cursor-changed` | 마우스 커서 종류가 바뀜 |

| `context-menu` | 컨텍스트 메뉴를 표시해야 함 (`params` 전달) |

```js filename="shortcuts.js"

    event.preventDefault() // Ctrl+R 새로고침 비활성화
  }
})
```

</EasyNote>

<Callout type="warning">

### 스크립트 & 스타일

| 메서드 | 하는 일 |
|---|---|
| `contents.executeJavaScript(code[, userGesture])` | 페이지에서 JS 실행; 결과의 `Promise` 반환 |

```js filename="inject.js"
await contents.removeInsertedCSS(key)
```

<Callout type="warning">
신뢰할 수 없는 문자열을 `executeJavaScript` 에 넘기지 마세요 — 페이지 권한으로 임의 코드를 실행하게 돼요. 코드는 직접 작성하고, 끼워 넣는 데이터는 반드시 `JSON.stringify` 하세요.

## 쉽게 풀어 설명

창에는 두 겹이 있는데, 입문자가 늘 헷갈려 해요:

1. **`BrowserWindow`** 은 _액자_ 예요 — 제목 표시줄, 최소화 버튼, 데스크톱 위 위치를 가진 OS 수준의 상자.
2. **`webContents`** 는 _페이지_ 예요 — 그 액자 안에 살아 있는 웹 문서로, URL·스크립트·스크롤 위치·콘솔을 가져요.

실제로 _하고 싶은 일_ 의 거의 전부 — "이 URL 로드", "이 스크립트 실행", "페이지 스크린샷", "로딩 끝났을 때 듣기", "페이지에 메시지 보내기" — 는 창이 아니라 `webContents` 에서 일어나요.

세 가지 패턴이면 실무의 대부분이 끝나요:

1. **기다린 뒤 행동.** `did-finish-load` 를 듣고 나서 `executeJavaScript` 하거나 `send` 로 데이터를 넣어요. (`dom-ready` 는 너무 일찍 발생하는 함정이에요.)
2. **단단히 잠그기.** `setWindowOpenHandler`(팝업 거부)와 `will-navigate`(신뢰 못 할 이동 차단)가 두 개의 안전벨트예요.
3. **페이지와 대화.** 나갈 땐 `contents.send(...)`, 들어올 땐 `contents.ipc.handle(...)` 와 `ipc-message` 이벤트예요.

기억하세요: `new WebContents()` 는 거의 쓰지 않아요. `win.webContents`, `webContents.fromId(id)`, 또는 `web-contents-created` 이벤트로 인스턴스를 받고 — 그 다음엔 그저 그 페이지 한 개의 리모컨일 뿐이에요.

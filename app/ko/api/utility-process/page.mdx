`utilityProcess` 는 **Main 프로세스** 에 살아요. Chromium 서비스 프로세스로부터 별도의 자식 프로세스를 만들어, 메인 프로세스나 창을 멈추게 하지 않고 CPU 를 많이 쓰거나 신뢰할 수 없는 Node.js 코드를 돌리기에 딱 좋아요.

```js filename="main.js"
const { utilityProcess } = require('electron')

// 스크립트 파일로부터 자식 프로세스를 생성해요
const child = utilityProcess.fork('worker.js')

child.on('spawn', () => {
  child.postMessage({ task: 'crunch-numbers', payload: [1, 2, 3] })
})

child.on('message', (result) => {
  // 워커가 무언가를 돌려보냈어요
})
```

<EasyNote>
`utilityProcess` 는 **한 가지 일만 맡길 프리랜서를 고용하는 것** 과 같아요. 메인 프로세스는 앱을 돌리느라 바쁘니까, 무거운 일을 직접 하려고 멈추는 대신 도우미를 한 명 띄우고, 작업을 우편함 하나(`postMessage`)로 건네준 뒤 같은 길로 답을 받아와요. 일이 끝나면 도우미는 집으로 돌아가요(`exit`).
</EasyNote>

## 프로세스 생성하기

`utilityProcess.fork()` 는 **팩토리 함수** 예요 — `new` 로 호출하지 않아요. `UtilityProcess` 인스턴스를 반환합니다.

| 메서드 | 하는 일 |
|---|---|
| `utilityProcess.fork(modulePath[, args][, options])` | 스크립트 파일로부터 자식 프로세스를 띄우고 `UtilityProcess` 를 반환 |

인자는 다음과 같아요.

| 인자 | 타입 | 의미 |
|---|---|---|
| `modulePath` | `string` | 자식 프로세스가 진입점으로 실행할 스크립트 경로 |
| `args` | `string[]` _(선택)_ | 추가 인자 — 자식에서 `process.argv` 로 접근 가능 |
| `options` | `Object` _(선택)_ | 자식 프로세스 설정 (아래 참고) |

### `options`

| 옵션 | 타입 | 하는 일 |
|---|---|---|
| `env` | `Object` | 자식의 환경 변수 (기본값 `process.env`) |
| `execArgv` | `string[]` | Node 실행 파일에 전달할 인자 |
| `cwd` | `string` | 자식 프로세스의 작업 디렉터리 |
| `stdio` | `string \| string[]` | stdin/stdout/stderr 설정: `'pipe'`, `'ignore'`, `'inherit'` |
| `serviceName` | `string` | `app.getAppMetrics()` 와 OS 작업 관리자에 표시될 프로세스 이름 |
| `session` | `Session` | 자식의 네트워크 요청에 사용할 `Session` |
| `partition` | `string` | 세션 파티션 문자열 (`session` 의 대안) |
| `allowLoadingUnsignedLibraries` | `boolean` _(macOS)_ | 서명되지 않은 동적 라이브러리 로딩 허용 |
| `disclaim` | `boolean` _(macOS)_ | TCC 책임을 부모로부터 상속하도록 포기 |
| `respondToAuthRequestsFromMainProcess` | `boolean` | HTTP 인증 요청을 `login` 이벤트로 메인 프로세스에 전달 |

```js filename="fork-options.js"
const { utilityProcess } = require('electron')
const path = require('path')

const child = utilityProcess.fork(path.join(__dirname, 'worker.js'), ['--mode=fast'], {
  serviceName: 'image-resizer',   // 작업 관리자에 표시돼요
  stdio: 'pipe',                  // 아래에서 stdout/stderr 를 캡처해요
  env: { ...process.env, WORKER: '1' },
})
```

<Callout type="warning">
자식의 진입점은 HTML 페이지가 아니라 Node.js 스크립트여야 해요 — `utilityProcess` 는 `BrowserWindow` 가 아니라 Node 환경을 실행해요. `stdout`/`stderr` 를 쓰려면 `stdio` 를 `'pipe'` 로 설정해야 하며, 그렇지 않으면 두 속성은 `null` 이 됩니다.
</Callout>

## 인스턴스 메서드(Instance Methods)

`fork()` 가 반환한 값에 대해 호출합니다.

| 메서드 | 하는 일 |
|---|---|
| `child.postMessage(message[, transfer])` | 자식 프로세스로 메시지를 보냄 — `transfer` 로 `MessagePortMain` 객체를 넘길 수 있음 |
| `child.kill()` | 자식 프로세스를 종료 — 종료 요청이 전달되면 `true` 반환 |

```js filename="message-channel.js"
const { utilityProcess, MessageChannelMain } = require('electron')

const child = utilityProcess.fork('worker.js')
const { port1, port2 } = new MessageChannelMain()

child.on('spawn', () => {
  // 양방향 통신을 위해 채널의 한쪽 끝을 자식에게 넘겨요
  child.postMessage({ message: 'hello' }, [port1])
})

port2.on('message', (event) => {
  // 워커가 보낸 포트 간 직접 메시지
})
port2.start()
```

```js filename="cleanup.js"
// 다 쓰고 나면 항상 워커를 정리해요
function shutdown(child) {
  if (child.pid !== undefined) {
    child.kill()
  }
}
```

<Callout type="warning">
`postMessage` 에 넘기는 `message` 는 [구조화 복제 알고리즘](https://developer.mozilla.org/ko/docs/Web/API/Web_Workers_API/Structured_clone_algorithm)으로 복제돼요. 그래서 함수나 DOM 노드는 담을 수 없어요. `transfer` 배열로 옮길 수 있는 건 `MessagePortMain` 인스턴스뿐이에요.
</Callout>

## 속성(Properties)

| 속성 | 타입 | 의미 |
|---|---|---|
| `child.pid` | `Integer \| undefined` (읽기 전용) | OS 프로세스 id — 프로세스가 생성되기 전에는 `undefined` |
| `child.stdout` | `NodeJS.ReadableStream \| null` (읽기 전용) | 자식의 표준 출력 — `stdio` 를 `'pipe'` 로 두지 않으면 `null` |
| `child.stderr` | `NodeJS.ReadableStream \| null` (읽기 전용) | 자식의 표준 에러 — `stdio` 를 `'pipe'` 로 두지 않으면 `null` |

```js filename="capture-output.js"
const child = utilityProcess.fork('worker.js', [], { stdio: 'pipe' })

// 워커의 로그를 메인 프로세스로 흘려보내요
child.stdout?.on('data', (data) => {
  process.stdout.write(`[worker] ${data}`)
})
child.stderr?.on('data', (data) => {
  process.stderr.write(`[worker:err] ${data}`)
})
```

## 이벤트(Events)

`child.on('이벤트명', 핸들러)` 로 듣습니다.

| 이벤트 | 발생 시점 |
|---|---|
| `spawn` | 자식 프로세스가 성공적으로 생성됨 — 이제 메시지를 보내도 안전 |
| `message` | 자식이 `process.parentPort.postMessage()` 로 메시지를 보냄 |
| `error` | 자식에서 V8 치명적 오류 발생 (`type`, `location`, 진단 리포트 전달) |
| `exit` | 프로세스가 종료됨 — 숫자 종료 `code` 전달 |
| `login` | 자식이 HTTP basic/proxy 인증이 필요함 (`respondToAuthRequestsFromMainProcess: true` 일 때만) |

```js filename="lifecycle.js"
const child = utilityProcess.fork('worker.js')

child.on('spawn', () => {
  // 프로세스가 살아 있어요 — 이제 말을 걸어도 안전해요
  child.postMessage({ start: true })
})

child.on('message', (data) => {
  // 워커가 결과를 우리에게 밀어 보냈어요
})

child.on('exit', (code) => {
  // 0 이면 정상 종료, 그 외면 크래시이거나 강제 종료된 거예요
  if (code !== 0) restartWorker()
})
```

<EasyNote>
생명주기는 전화 통화처럼 흘러가요. `spawn` 은 "상대가 받았다", `message` 는 "상대가 무언가 말했다", `error` 는 "크래시로 통화가 끊겼다", `exit` 는 "상대가 전화를 끊었다" 예요. 말을 걸기 전에 꼭 `spawn` 을 기다리세요 — 너무 일찍 보낸 메시지는 그냥 바닥에 흘려버려요.
</EasyNote>

```js filename="auth.js"
// 자식의 HTTP 인증 요청을 메인 프로세스에서 처리해요
const child = utilityProcess.fork('fetcher.js', [], {
  respondToAuthRequestsFromMainProcess: true,
})

child.on('login', (event, authInfo, callback) => {
  event.preventDefault()
  callback('username', 'password')
})
```

## 워커 쪽 코드

`fork` 한 스크립트 안에서는 `process.parentPort` 를 통해 부모와 대화해요.

```js filename="worker.js"
// 이 파일은 유틸리티(자식) 프로세스에서 실행돼요 — 평범한 Node.js 예요
process.parentPort.on('message', (event) => {
  const { task, payload } = event.data

  if (task === 'crunch-numbers') {
    const total = payload.reduce((a, b) => a + b, 0)
    // 답을 메인 프로세스로 돌려보내요
    process.parentPort.postMessage({ total })
  }
})
```

<Callout type="warning">
자식 쪽에서는 메시지 객체가 첫 번째 인자가 아니라 `event.data` 로 전달돼요. `.data` 를 빼먹는 게 `utilityProcess` 에서 가장 흔한 실수예요.
</Callout>

## 쉽게 풀어 설명

`utilityProcess` 는 단 하나의 질문에 답하려고 존재해요. **"무겁거나 위험한 작업을 어디에 둘까?"**

메인 프로세스는 반응이 빨라야 해요 — 창, 메뉴, 앱 생명주기를 관리하니까요. 거대한 계산을 돌리거나, 신뢰할 수 없는 파일을 파싱하거나, 네이티브 모듈을 거기서 로드하면 전부 멈춰버려요. 그래서 그 일을 도우미에게 맡기는 거예요. `utilityProcess.fork('worker.js')` 는 그 한 가지 작업만 전담하는 새로운 Node.js 프로세스를 시작해요.

둘은 깔끔한 채널 하나로 대화해요. 부모는 `child.postMessage(...)` 로 보내고 `child.on('message', ...)` 로 듣고, 자식은 `process.parentPort.postMessage(...)` 로 보내고 `process.parentPort.on('message', ...)` 로 들어요. 공유 메모리도, 깜짝 놀랄 일도 없이 — 브라우저의 Web Worker 처럼 메시지만 주고받아요.

이름 세 개만 기억하면 돼요. **`fork`**(도우미 시작), **`postMessage`**(말 걸기), **`exit`**(언제 떠났는지 알기). 첫 메시지 전에 **`spawn`** 을 기다리고, 로그가 필요하면 **`stdio: 'pipe'`** 를 켜고, 다 쓰면 **`kill()`** 을 부르세요.

`MessageChannelMain` 은 **Main 프로세스** 에서 `new` 로 만드는 **클래스** 예요. DOM 의 [`MessageChannel`](https://developer.mozilla.org/ko/docs/Web/API/MessageChannel) 객체에 해당하는 Main 프로세스 버전이죠. 하는 일은 딱 하나 — 서로 연결된 [`MessagePortMain`](#messageportmain) 객체 한 쌍(`port1`, `port2`)을 만드는 거예요. 이 두 포트를 각각 다른 곳에 보내면, 그 사이를 잇는 직통 채널이 생겨요.

```js filename="main.js"
const { BrowserWindow, MessageChannelMain } = require('electron')

const w = new BrowserWindow()

// 채널 하나가 연결된 두 끝을 만들어줘요
const { port1, port2 } = new MessageChannelMain()

// 한쪽 끝은 렌더러에 건네고, 다른 쪽은 Main 에 남겨요
w.webContents.postMessage('port', null, [port2])

// port1 에 보낸 건 port2 에 도착하고, 그 반대도 마찬가지예요
port1.postMessage({ some: 'message' })
```

<EasyNote>
`MessageChannelMain` 은 **한 세트로 파는 무전기** 같아요. 상자를 열면(`new MessageChannelMain()`) 무전기 두 대 `port1`, `port2` 가 나와요. 이미 같은 전용 주파수로 맞춰져 있죠. 하나는 내가 쥐고 다른 하나는 친구(렌더러, 워커)에게 주면, 한쪽이 말하는 걸 다른 쪽이 그대로 들어요 — 그 채널엔 우리 둘뿐이에요.
</EasyNote>

---

## 생성자(Constructor)

| 생성자 | 하는 일 |
|---|---|
| `new MessageChannelMain()` | 새 채널을 만들고, 서로 연결된 두 포트 `port1`, `port2` 를 담은 객체를 반환 |

두 포트는 한 몸처럼 얽혀 있어요 — 한쪽에 보낸 메시지가 다른 쪽에서 수신돼요. 전달할 옵션은 없어요.

---

## 인스턴스 속성(Instance properties)

`MessageChannelMain` 인스턴스에는 속성이 딱 두 개 — 채널의 양 끝이에요.

| 속성 | 타입 | 무엇인가 |
|---|---|---|
| `channel.port1` | `MessagePortMain` (읽기전용) | 채널의 한쪽 끝 |
| `channel.port2` | `MessagePortMain` (읽기전용) | `port1` 과 연결된 반대쪽 끝 |

```js filename="ports.js"
const { port1, port2 } = new MessageChannelMain()

// 한쪽에 하는 일은 다른 쪽에 그대로 비춰져요 —
// 같은 전용 회선의 양 끝이니까요.
```

<Callout type="info">
`MessageChannelMain` 자체에는 **이벤트도 메서드도 없어요** — 두 포트를 만들어주는 공장 역할만 해요. 실제 일은 아래의 `MessagePortMain` 객체에서 일어나요.
</Callout>

## MessagePortMain

`MessagePortMain` 은 DOM 의 [`MessagePort`](https://developer.mozilla.org/ko/docs/Web/API/MessagePort) 에 해당하는 Main 프로세스 버전이에요. 직접 생성하지 않고, `MessageChannelMain` 에서 (`port1` / `port2` 로) 얻거나, 도착한 메시지에 딸려 온 `ports` 로 받아요. Node 의 `EventEmitter` 를 상속해요.

### 인스턴스 이벤트(Instance events)

`port.on('이벤트명', 핸들러)` 로 듣습니다.

| 이벤트 | 발생 시점 | 전달값 |
|---|---|---|
| `message` | 반대쪽 끝에서 보낸 메시지를 포트가 받음 | `data`(`any`) 와 `ports`(`MessagePortMain[]`) 를 가진 이벤트 객체 |
| `close` | 포트의 원격(반대쪽) 끝이 연결 해제됨 | _(없음)_ |

```js filename="receive.js"
const { port1, port2 } = new MessageChannelMain()

port2.on('message', (event) => {
  // event.data 는 반대쪽이 보낸 그 내용이에요
  handle(event.data)
  // event.ports 에는 함께 전달된 MessagePortMain 객체들이 들어 있어요
})

port2.on('close', () => {
  // 반대쪽이 사라졌어요 — 이 포트 사용을 멈춰요
  cleanup()
})

// 큐에 쌓인 메시지 전달을 시작해요 (아래 경고 참고)
port2.start()
```

<EasyNote>
`message` 이벤트는 무전기의 **스피커**, `postMessage` 는 **마이크** 예요. `close` 는 상대가 통화를 끊을 때 들리는 작은 "딸깍" 소리고요 — 끊긴 회선에 더 말하지 말라는 신호죠.
</EasyNote>

### 인스턴스 메서드(Instance methods)

| 메서드 | 하는 일 |
|---|---|
| `port.postMessage(message[, transfer])` | 반대쪽 끝으로 `message` 전송. 선택적으로 `transfer`(`MessagePortMain[]` 배열)를 넘기면 그 포트들의 소유권을 상대에게 넘김 |
| `port.start()` | 메시지 전달 시작. 이 메서드를 부르기 전까지 수신 메시지는 **큐에 쌓임** |
| `port.close()` | 포트 연결을 끊어 더 이상 활성 상태가 아니게 함 |

```js filename="transfer.js"
// 기존 포트를 *통해* 새 포트를 보낼 수 있어요 —
// 프로세스들이 직접 대화하는 그물망을 이렇게 만들어요.
const { port1, port2 } = new MessageChannelMain()
const sub = new MessageChannelMain()

// port1 반대쪽에 있는 상대에게 sub.port2 를 보내요
port1.postMessage({ kind: 'new-channel' }, [sub.port2])

// 이제 (여기 남긴) sub.port1 은 sub.port2 의 수신자와 직접 대화해요
sub.port1.postMessage('서브 채널로 보내는 인사')
```

<Callout type="warning">
메시지는 **`port.start()` 를 부를 때까지 큐에 쌓여요** (렌더러 쪽에서는 `onmessage` 핸들러를 할당하면 자동으로 `start()` 가 호출돼요). `port.on('message', ...)` 로 리스너만 달고 `port.start()` 를 부르지 않으면, 핸들러는 영영 실행되지 않고 메시지가 조용히 쌓이기만 해요.
</Callout>

<Callout type="warning">
`MessagePortMain` 객체는 **복사가 아니라 이전(transfer)** 돼요. `postMessage` 의 `transfer` 배열(또는 `webContents.postMessage`)로 포트를 한 번 넘기면, 보낸 쪽은 더 이상 그 포트를 쓸 수 없어요 — 소유권이 받는 쪽으로 옮겨가요. 한쪽은 남기고 다른 쪽만 보내세요. 두 복사본을 동시에 쓰려 하지 마세요.
</Callout>

### 인스턴스 속성(Instance properties)

`MessagePortMain` 에는 문서화된 인스턴스 속성이 없어요 — 위의 이벤트와 메서드로만 다뤄요.

## 양쪽은 어떻게 연결될까

`MessageChannelMain` 의 핵심은 Main 프로세스와 렌더러(또는 렌더러 둘)를 직통 회선으로 잇는 거예요. 모든 메시지를 `ipcMain`/`ipcRenderer` 로 돌리지 않고요. Main 쪽은 `MessagePortMain` 을, 렌더러 쪽은 평범한 DOM `MessagePort` 를 받아요.

```js filename="renderer.js"
const { ipcRenderer } = require('electron')

ipcRenderer.on('port', (event) => {
  // event.ports 는 이 메시지와 함께 보내진 포트 목록이에요
  const port = event.ports[0]

  // 렌더러에서는 onmessage 를 할당하면 start() 가 자동으로 불려요
  port.onmessage = (messageEvent) => {
    document.title = messageEvent.data.some
  }

  // 렌더러는 같은 회선으로 곧장 답장할 수 있어요
  port.postMessage('렌더러에서 보내는 인사')
})
```

<Callout type="info">
**렌더러** 쪽 포트는 표준 웹 `MessagePort` 라서 `port.onmessage = ...` / `port.addEventListener('message', ...)` 와 `port.start()` 를 써요. **Main** 쪽은 `MessagePortMain` 이라서 `port.on('message', ...)` 와 `port.start()` 를 써요. 두 API 모양은 일부러 닮게 만들어졌어요.
</Callout>

## 쉽게 풀어 설명

`MessageChannelMain` 은 작지만 큰 아이디어를 담은 클래스예요 — **앱의 두 부분을 잇는 전용 직통 통로** 죠.

보통 렌더러는 공용 `ipcRenderer` ↔ `ipcMain` 교환대를 거쳐 Main 프로세스와 대화해요. 그래도 충분하지만, 모든 메시지가 같은 교환원을 거쳐요. 때로는 전용 회선이 필요하죠 — 예를 들어 렌더러 창 둘이 서로 직접 대화하게 하거나, 바쁜 워커 창이 주 IPC 트래픽을 어지럽히지 않는 자기만의 채널을 갖게 하고 싶을 때요.

전체 이야기는 세 박자예요:

1. **상자를 연다.** `new MessageChannelMain()` 이 연결된 포트 두 개 `port1`, `port2` 를 줘요. 이미 이어져 있어서, 한쪽이 보내면 다른 쪽이 받아요.
2. **한쪽 끝을 건넨다.** `webContents.postMessage('채널이름', null, [port2])` 로 필요한 곳에 포트 하나를 보내요. 보낸 쪽은 그 포트를 잃고(*복사*가 아니라 *이전*), 받는 쪽이 갖게 돼요.
3. **대화한다.** `port.postMessage(data)` 로 보내고, `message` 이벤트로 받아요. Main 쪽에서는 먼저 `port.start()` 를 부르고, `close` 에서 정리하면 돼요.

세 이름만 외우면 나머지는 찾아보면 돼요: **`new MessageChannelMain()`**(포트 두 개 얻기), **`postMessage`**(보내기), **`start()`**(받기 시작). 교환원 없는, 점대점 직통 채널이에요.

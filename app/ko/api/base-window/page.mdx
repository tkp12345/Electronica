`BaseWindow` 는 **Main 프로세스** 에서 `new` 로 만드는 **클래스** 예요. 실제 OS 창이지만, `BrowserWindow` 와 달리 **자기 자신의 웹 페이지가 없어요**. 대신 `contentView` 에 하나 이상의 `View`(예: `WebContentsView`)를 붙여서 화면을 채워요. 여러 페이지를 나란히 구성하는 창을 만들 때의 기본 빌딩 블록이에요.

```js filename="main.js"
const { BaseWindow, WebContentsView } = require('electron')

const win = new BaseWindow({ width: 800, height: 600 })

// BaseWindow 는 단일 페이지가 아니라 여러 뷰를 담아요
const view = new WebContentsView()
win.contentView.addChildView(view)
view.setBounds({ x: 0, y: 0, width: 800, height: 600 })
view.webContents.loadURL('https://electronjs.org')
```

<EasyNote>
`BaseWindow` 를 **속에 코르크보드가 들어 있는 빈 액자** 라고 생각해 보세요. 액자(프레임)는 크기·위치·창 상태를 다루고, 코르크보드(`contentView`)는 사진(`View`)을 한 장 또는 여러 장 꽂아 두는 자리예요. `BrowserWindow` 는 사진 한 장이 이미 꽂힌 채로 나오는 `BaseWindow` 일 뿐이에요.
</EasyNote>

## 정적 메서드(Static methods)

인스턴스가 아니라 클래스 자체에서 호출해요.

| 메서드 | 하는 일 |
|---|---|
| `BaseWindow.getAllWindows()` | 열려 있는 모든 창을 `BaseWindow[]` 로 반환 |
| `BaseWindow.getFocusedWindow()` | 현재 포커스된 창, 없으면 `null` |
| `BaseWindow.fromId(id)` | 주어진 숫자 `id` 의 창, 없으면 `null` |

```js filename="reopen.js"
// macOS: 열린 창이 없을 때 Dock 아이콘을 누르면 창을 다시 만들어요
app.on('activate', () => {
  if (BaseWindow.getAllWindows().length === 0) createWindow()
})
```

## 인스턴스 이벤트(Instance events)

`win.on('이벤트명', 핸들러)` 로 듣습니다. _(macOS)_ / _(Windows)_ / _(Linux)_ 표시는 해당 플랫폼에서만 발생해요.

### 생명주기 & 표시

| 이벤트 | 발생 시점 |
|---|---|
| `close` | 창이 닫히려는 시점 — `preventDefault()` 로 취소 가능 |
| `closed` | 창이 닫힘 — 참조를 버리세요, 이미 사라졌어요 |
| `show` | 창이 표시됨 |
| `hide` | 창이 숨겨짐 |
| `focus` | 창이 포커스를 얻음 |
| `blur` | 창이 포커스를 잃음 |
| `query-session-end` _(Windows)_ | OS 세션이 끝나려는 시점 — `preventDefault()` 로 지연 가능 |
| `session-end` _(Windows)_ | OS 세션 종료 중(로그아웃/종료) — 막을 수 없음 |

```js filename="confirm-close.js"
win.on('close', (event) => {
  if (hasUnsavedWork()) {
    event.preventDefault() // 창을 열어 둬요
    showSaveDialog()
  }
})

win.on('closed', () => {
  // 창이 사라졌어요 — GC 되도록 참조를 비워요
  win = null
})
```

<Callout type="warning">
`closed` 가 발생한 뒤에는 창이 파괴된 상태예요. 그 창의 어떤 메서드를 호출해도 예외가 발생해요. `closed` 핸들러 안에서 항상 참조를 `null` 로 비워 주세요.
</Callout>

### 크기 · 위치 · 상태

| 이벤트 | 발생 시점 |
|---|---|
| `maximize` | 창이 최대화됨 |
| `unmaximize` | 창이 최대화 상태에서 벗어남 |
| `minimize` | 창이 최소화됨 |
| `restore` | 최소화에서 복원됨 |
| `will-resize` _(macOS/Windows)_ | 크기 변경 직전 — `preventDefault()` 로 차단 가능 |
| `resize` | 크기 변경 후 |
| `resized` _(macOS/Windows)_ | 크기 변경 제스처가 끝남 |
| `will-move` _(macOS/Windows)_ | 이동 직전 — `preventDefault()` 로 차단 가능 |
| `move` | 창이 이동 중 |
| `moved` _(macOS/Windows)_ | 이동 제스처가 끝남 |
| `always-on-top-changed` | 항상 위 플래그가 토글됨 |

### 전체화면

| 이벤트 | 발생 시점 |
|---|---|
| `enter-full-screen` | 전체화면 진입 |
| `leave-full-screen` | 전체화면 해제 |

### 제스처 & 플랫폼 부가 기능

| 이벤트 | 발생 시점 |
|---|---|
| `app-command` _(Windows/Linux)_ | App Command 호출됨 (예: 미디어 키, 브라우저 뒤로) |
| `swipe` _(macOS)_ | 세 손가락 트랙패드 스와이프 |
| `rotate-gesture` _(macOS)_ | 트랙패드 회전 제스처 |
| `sheet-begin` _(macOS)_ | 창 시트가 열림 |
| `sheet-end` _(macOS)_ | 창 시트가 닫힘 |
| `new-window-for-tab` _(macOS)_ | 네이티브 "새 탭" 버튼 클릭 |
| `system-context-menu` _(Windows/Linux)_ | 시스템 컨텍스트 메뉴 트리거 — `preventDefault()` 로 억제 가능 |

<EasyNote>
`BrowserWindow` 와 비교해 **무엇이 없는지** 눈여겨보세요. `ready-to-show` 도, `page-title-updated` 도, `responsive` / `unresponsive` 도 없어요. 그것들은 _웹 페이지_ 에 관한 이벤트인데, `BaseWindow` 는 자기 페이지가 없거든요. 필요하면 자식 뷰의 `webContents` 에서 들으세요.
</EasyNote>

## 인스턴스 메서드(Instance methods)

### 생명주기

| 메서드 | 하는 일 |
|---|---|
| `win.destroy()` | `close` 를 발생시키지 **않고** 강제로 창 닫기 |
| `win.close()` | 정상적으로 닫기 (`close` 발생, 취소 가능) |
| `win.focus()` / `win.blur()` | 창 포커스 주기 / 해제 |
| `win.isFocused()` | 이 창이 포커스되어 있나? |
| `win.isDestroyed()` | 창이 파괴되었으면 `true` |

### 콘텐츠 뷰

| 메서드 | 하는 일 |
|---|---|
| `win.setContentView(view)` | 창 콘텐츠 영역을 채우는 루트 `View` 설정 |
| `win.getContentView()` | 루트 콘텐츠 `View` 가져오기 |
| `win.setAspectRatio(ratio[, extraSize])` | 크기 조절 종횡비 고정 (`extraSize` _(macOS)_) |

```js filename="multi-view.js"
const { BaseWindow, WebContentsView } = require('electron')

const win = new BaseWindow({ width: 1000, height: 600 })

// 한 창 안에 두 페이지를 나란히 구성해요
const left = new WebContentsView()
const right = new WebContentsView()
win.contentView.addChildView(left)
win.contentView.addChildView(right)

left.setBounds({ x: 0, y: 0, width: 500, height: 600 })
right.setBounds({ x: 500, y: 0, width: 500, height: 600 })

left.webContents.loadURL('https://electronjs.org')
right.webContents.loadFile('sidebar.html')
```

<Callout type="warning">
`BaseWindow` 에는 `loadURL` / `loadFile` 도, `webContents` 도 **없어요**. 페이지를 창에 직접 로드할 수 없고, `WebContentsView` 를 붙인 뒤 그 _뷰_ 의 `webContents` 에 로드해야 해요. 창에 페이지 하나만 띄우고 싶다면 `BrowserWindow` 를 쓰세요.
</Callout>

## 쉽게 풀어 설명

`BaseWindow` 는 **웹 페이지가 없는 창** 이에요. `BrowserWindow` 가 "브라우저 하나가 들어 있는 창" 이라면, `BaseWindow` 는 "직접 채워야 하는 빈 콘텐츠 영역이 있는 창" 이에요. 이 한 가지 차이가 이 페이지의 모든 것을 설명해요.

1. **프레임은 똑같아요.** 크기, 위치, 최대화/최소화, 전체화면, 항상 위, 제목 표시줄, 작업표시줄 — 이 모든 게 `BrowserWindow` 와 정확히 같게 동작해요. 사실 `BrowserWindow` 가 `BaseWindow` 를 _상속_ 하거든요.
2. **속이 달라요.** `loadURL` 도, `webContents` 도 없어요. 대신 `contentView` 가 있고, 거기에 `View` 를 붙여요 — 대개 `WebContentsView`(페이지 하나) 하나, 또는 여러 개를 나란히 배치하죠.

그럼 언제 쓰냐고요? **한 창 안에 웹 페이지가 둘 이상 필요할 때** 예요 — 사이드바와 메인 패널, 브라우저식 멀티탭 레이아웃, 콘텐츠 위에 얹는 오버레이처럼요. 그 외에는 페이지 하나짜리 지름길인 `BrowserWindow` 가 더 편해요.

거래 조건을 기억하세요: `BaseWindow` 는 직접 페이지를 연결하는 수고를 대가로 조합 능력(**`contentView` + `addChildView`**)을 주고, `BrowserWindow` 는 페이지가 하나뿐이라는 제약을 대가로 페이지를 공짜로 줘요.

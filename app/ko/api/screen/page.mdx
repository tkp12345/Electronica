`screen` 모듈은 **Main 프로세스** 에 살아요. `new` 로 만들지 않는 `EventEmitter` 모듈이며, 연결된 모든 디스플레이의 크기·배치·배율과 실시간 커서 위치를 알려줘요.

```js filename="main.js"
const { app, screen } = require('electron')

app.whenReady().then(() => {
  const primary = screen.getPrimaryDisplay()
  const { width, height } = primary.workAreaSize
  // 메인 모니터의 사용 가능 영역에 맞춰 창을 만들어요
})
```

<Callout type="warning">
`screen` 은 `app` 모듈의 `ready` 이벤트가 발생하기 전에는 사용할 수 없어요. 그 전에(스크립트 최상단에서) 읽으면 쓸모 있는 값이 안 나오고 에러가 날 수도 있어요.
</Callout>

<EasyNote>
`Display` 는 모니터 하나를 설명하는 평범한 객체예요. `id`, `bounds`(전체 사각형), `workArea`(작업표시줄·메뉴 막대를 뺀 영역), `scaleFactor`(일반 화면은 1, Retina/HiDPI 는 2), `rotation` 등을 담고 있어요. 여기 나오는 메서드들은 모두 이 객체 하나, 그 목록, 또는 `{ x, y }` 점을 돌려줘요.
</EasyNote>

## 이벤트(Events)

`screen.on('이벤트명', 핸들러)` 로 듣습니다. 모두 사용자의 모니터 구성이 바뀔 때 발생해요.

| 이벤트 | 발생 시점 |
|---|---|
| `display-added` | 새 디스플레이가 연결됨 — 핸들러가 `(event, newDisplay)` 를 받음 |
| `display-removed` | 디스플레이가 연결 해제됨 — 핸들러가 `(event, oldDisplay)` 를 받음 |
| `display-metrics-changed` | 디스플레이 메트릭이 바뀜 — 핸들러가 `(event, display, changedMetrics)` 를 받으며, `changedMetrics` 는 `'bounds'`, `'workArea'`, `'scaleFactor'`, `'rotation'` 으로 이루어진 문자열 배열 |

```js filename="watch-displays.js"
app.whenReady().then(() => {
  screen.on('display-added', (event, newDisplay) => {
    // 모니터가 연결됨 — 창을 그쪽으로 옮길 수도 있어요
  })

  screen.on('display-removed', (event, oldDisplay) => {
    // 모니터가 분리됨 — 그 화면에 있던 창을 구출하세요
    rescueWindowsFrom(oldDisplay)
  })

  screen.on('display-metrics-changed', (event, display, changedMetrics) => {
    if (changedMetrics.includes('scaleFactor')) {
      // 사용자가 DPI/배율을 바꿈 — 픽셀 크기를 캐싱했다면 다시 배치하세요
    }
  })
})
```

<EasyNote>
이 세 이벤트를 모니터의 "연결 / 분리 / 재배치" 알림이라고 생각하세요. 앱이 창 위치를 기억한다면, `display-removed` 가 바로 아직 존재하는 화면으로 창을 다시 끌어올 신호예요 — 안 그러면 창이 화면 밖에 갇힐 수 있어요.
</EasyNote>

## 메서드(Methods)

### 디스플레이

| 메서드 | 하는 일 |
|---|---|
| `screen.getPrimaryDisplay()` | 주 `Display`(메뉴 막대/메인 작업표시줄이 있는 화면) |
| `screen.getAllDisplays()` | 현재 연결된 모든 모니터의 `Display[]` |
| `screen.getDisplayNearestPoint(point)` | 주어진 `{ x, y }` 점에 가장 가까운 `Display` |
| `screen.getDisplayMatching(rect)` | 주어진 `Rectangle` 와 가장 많이 겹치는 `Display` |

```js filename="window-on-cursor.js"
// 마우스가 지금 있는 모니터에 창을 엽니다
const cursor = screen.getCursorScreenPoint()
const display = screen.getDisplayNearestPoint(cursor)
const { x, y, width, height } = display.workArea

const win = new BrowserWindow({
  x: x + (width - 800) / 2,
  y: y + (height - 600) / 2,
  width: 800,
  height: 600,
})
```

### 커서

| 메서드 | 하는 일 |
|---|---|
| `screen.getCursorScreenPoint()` | 현재 마우스의 절대 위치를 DIP `{ x, y }` 점으로 반환 _(Wayland 제외)_ |

### 좌표 변환 _(Windows/Linux)_

**물리** 픽셀(하드웨어가 쓰는 단위)과 **DIP**(밀도 독립 픽셀, 레이아웃이 쓰는 단위) 사이를 변환해요. macOS 에서는 둘이 이미 일치하므로 이 메서드들이 필요 없어요.

| 메서드 | 플랫폼 | 하는 일 |
|---|---|---|
| `screen.screenToDipPoint(point)` | Windows, Linux | 물리 화면 점 → DIP 점 _(Wayland 제외)_ |
| `screen.dipToScreenPoint(point)` | Windows, Linux | DIP 점 → 물리 화면 점 _(Wayland 제외)_ |
| `screen.screenToDipRect(window, rect)` | Windows | 물리 `Rectangle` → DIP `Rectangle` (`window` 은 `null` 가능) |
| `screen.dipToScreenRect(window, rect)` | Windows | DIP `Rectangle` → 물리 `Rectangle` (`window` 은 `null` 가능) |

```js filename="dip-conversion.js"
// Windows/Linux: 원시 OS 픽셀 좌표를 렌더러와 BrowserWindow API 가
// 기대하는 DIP 공간으로 변환해요
const dip = screen.screenToDipPoint({ x: 2400, y: 100 })
```

<Callout type="warning">
`getCursorScreenPoint`, `screenToDipPoint`, `dipToScreenPoint` 는 **Wayland(Linux) 에서 지원되지 않아요.** 플랫폼별 호출은 방어 코드로 감싸고, 변환 메서드가 macOS 에 존재한다고 가정하지 마세요 — macOS 에서는 물리·DIP 좌표가 동일해서 아예 없어요.
</Callout>

## 속성(Properties)

`screen` 모듈에는 **속성이 없어요** — 모든 것은 위의 메서드와 이벤트로 노출돼요. (메서드가 돌려주는 각 `Display` 객체는 `id`, `bounds`, `workArea`, `size`, `workAreaSize`, `scaleFactor`, `rotation`, `internal`, `touchSupport` 등 자체 필드를 가져요.)

## 쉽게 풀어 설명

`screen` 모듈은 앱이 보는 **사용자 책상의 지도** 예요. 누군가는 노트북 화면 하나만, 누군가는 모니터 세 대를 나란히, 하나는 세로로 돌려서, 서로 다른 해상도로 쓰죠. `screen` 은 바로 그 배치를 알려줘서 창이 적절한 자리에 놓이게 해요.

세 가지면 거의 다 돼요:

1. **"창을 어디 둘까?"** → 메인 모니터는 `getPrimaryDisplay()`, 전부 보려면 `getAllDisplays()`. 작업표시줄 밑에 창이 깔리지 않도록 `bounds` 가 아니라 디스플레이의 `workArea` 를 쓰세요.
2. **"마우스는 어디 있지?"** → `getCursorScreenPoint()`, 흔히 `getDisplayNearestPoint()` 와 함께 써서 사용자가 실제로 보고 있는 화면에 창을 열어요.
3. **"구성이 바뀌었나?"** → `display-added` / `display-removed` / `display-metrics-changed` 이벤트. 창 위치를 기억한다면 이게 바로 그 위치가 아직 유효한지 다시 확인할 신호예요.

무엇보다 한 가지 규칙: `screen` 을 건드리기 전에 `app` 이 **`ready`** 될 때까지 기다리세요. 이것만 지키고, `workArea` 와 `bounds` 의 차이만 기억하면 나머지는 찾아보면 돼요.

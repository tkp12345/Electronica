`contentTracing` 모듈은 **Main 프로세스** 에 살아요. Chromium 에 내장된 추적(tracing) 시스템에 연결해서, 메인·렌더러·GPU·유틸리티 등 모든 프로세스가 무슨 일을 했는지 상세한 타임라인으로 수집해 줍니다. 덕분에 앱에서 느린 부분을 콕 집어낼 수 있어요.

```js filename="main.js"
const { app, contentTracing } = require('electron')

app.whenReady().then(async () => {
  // 5초 동안 전부 기록한 다음, 추적 결과를 파일로 저장합니다
  await contentTracing.startRecording({ included_categories: ['*'] })
  await new Promise((resolve) => setTimeout(resolve, 5000))
  const path = await contentTracing.stopRecording()
  // 저장된 `path` 를 chrome://tracing 이나 Perfetto UI 에서 열어 분석합니다
})
```

<EasyNote>
`contentTracing` 은 앱의 **블랙박스(비행 기록 장치)** 예요. 녹화 버튼을 누르고, 앱을 돌리는 동안 내부에서 일어나는 모든 동작을 마이크로초 단위 시각과 함께 기록한 뒤, 정지 버튼을 누르고 저장된 테이프를 뷰어로 읽어 보는 거죠. 스스로 뭔가를 고쳐 주지는 않아요. 다만 시간이 어디로 샜는지를 정확히 알려 줍니다.
</EasyNote>

<Callout type="info">
결과 파일은 Chromium 추적 파일이에요. 아무 Chromium 브라우저에서 `chrome://tracing` 으로 열거나, [Perfetto UI](https://ui.perfetto.dev) 에 끌어다 놓으면 더 풍부한 타임라인으로 볼 수 있어요.
</Callout>

## 이벤트(Events)

`contentTracing` 에는 **이벤트가 없어요**. 순수한 명령형 모듈이라서, 메서드를 호출하고 그 메서드가 돌려주는 Promise 를 `await` 할 뿐 — `.on(...)` 으로 구독할 대상이 없습니다.

---

## 메서드(Methods)

모든 메서드는 Promise 를 반환하며 **Main 프로세스** 에서만 동작해요. 자식 프로세스 전체를 조율하기 때문에, 모든 프로세스가 응답을 보내야 호출이 완료(resolve)됩니다.

### 기록 (Recording)

| 메서드 | 하는 일 |
|---|---|
| `contentTracing.startRecording(options)` | 모든 프로세스에서 기록을 시작합니다. 모든 곳에서 기록이 시작되면 완료되는 `Promise<void>` 반환. `options` 는 `TraceConfig`(예: `included_categories`) 또는 예전 방식인 `TraceCategoriesAndOptions` |
| `contentTracing.stopRecording([resultFilePath])` | 모든 프로세스에서 기록을 멈춥니다. 추적 데이터가 기록된 파일 경로로 완료되는 `Promise<string>` 반환. `resultFilePath` 를 생략하면 임시 파일에 기록됩니다 |

```js filename="record.js"
// 전부 대신 몇 개 카테고리만 추적해서 잡음을 줄입니다
await contentTracing.startRecording({
  included_categories: ['v8', 'blink'],
})

// ... 측정하고 싶은 느린 흐름을 실행 ...

// 나중에 찾기 쉽도록 특정 경로에 저장합니다
const file = await contentTracing.stopRecording('/tmp/my-trace.json')
```

<Callout type="warning">
추적 작업은 한 번에 **하나만** 실행할 수 있어요. `startRecording()` 한 번에는 반드시 `stopRecording()` 한 번을 짝지어 주세요. 멈추지 않고 `startRecording()` 을 두 번 호출하거나, 기록 중이 아닐 때 `stopRecording()` 을 호출하면 reject 됩니다.
</Callout>

### 살펴보기 (Inspecting)

| 메서드 | 하는 일 |
|---|---|
| `contentTracing.getCategories()` | 추적할 수 있는 카테고리 그룹 목록을 가져옵니다. `Promise<string[]>` 반환. 코드 경로가 실행될수록 목록이 늘어나므로, 아직 한 번도 실행되지 않은 카테고리는 보이지 않을 수 있어요 |
| `contentTracing.getTraceBufferUsage()` | 모든 프로세스를 통틀어 추적 버퍼의 최대 사용량을 가져옵니다. `value`(숫자) 와 `percentage`(버퍼가 찬 비율, 0~100) 를 담은 `Promise<Object>` 반환 |

```js filename="inspect.js"
// 시작하기 전에 무엇을 기록할 수 있는지 먼저 확인합니다
const categories = await contentTracing.getCategories()

// 기록 중에 버퍼가 얼마나 찼는지 확인 — 100% 가 되면 데이터를 잃기 시작합니다
const { value, percentage } = await contentTracing.getTraceBufferUsage()
if (percentage > 90) {
  // 버퍼가 거의 찼어요 — 곧 멈추지 않으면 이벤트가 누락됩니다
  await contentTracing.stopRecording('/tmp/trace.json')
}
```

<EasyNote>
추적 버퍼는 **저장 공간이 한정된 블랙박스 카메라** 와 같아요. 계속 녹화하지만, 가득 차면 가장 오래된 영상부터 덮어써집니다. `getTraceBufferUsage()` 는 "저장 공간 거의 가득 참" 경고등 같은 거라, 무언가 사라지기 전에 멈추고 저장하라고 알려 줘요.
</EasyNote>

### 실험적 기능 (Experimental)

| 메서드 | 하는 일 |
|---|---|
| `contentTracing.enableHeapProfiling([options])` _(실험적)_ | 힙 메모리 프로파일링을 켜서 `memory-infra` 추적에 메모리 데이터가 포함되도록 합니다. `Promise<void>` 반환. 반드시 `startRecording()` **이전에** 호출해야 해요 |

<Callout type="warning">
`enableHeapProfiling()` 은 **실험적** 기능이에요 — 동작과 시그니처가 Electron 버전마다 바뀔 수 있습니다. 진단 용도로만 쓰고, 배포되는 프로덕션 코드 경로에서는 사용하지 마세요.
</Callout>

## 속성(Properties)

`contentTracing` 에는 **속성이 없어요**. 상태가 없는 명령형 인터페이스라서, 필요한 모든 정보는 메서드가 돌려주는 Promise 를 통해 받습니다.

---

## 쉽게 풀어 설명

`contentTracing` 은 단 하나의 질문에 답해요. **"왜 바로 여기서 내 앱이 느릴까?"**

전체 작업 흐름은 호출 두 번이 전부예요. **`startRecording()`** 을 누르고, 느리게 느껴지는 바로 그 동작을 실행한 뒤, **`stopRecording()`** 을 누르면 파일 경로를 돌려받습니다. 그 파일을 `chrome://tracing` 이나 Perfetto 에서 열면, Electron 의 모든 프로세스에 걸쳐 일어난 함수 호출·렌더링·네트워크 대기를 타임라인 위에 펼쳐서 _눈으로_ 볼 수 있어요 — 길게 늘어진 막대가 바로 병목이고요.

나머지 세 메서드는 이 핵심 루프를 돕는 도구예요. **`getCategories()`** 는 무엇을 기록할 수 있는지 알려 주고, **`getTraceBufferUsage()`** 는 기록 버퍼가 넘치기 전에 경고해 주며, **`enableHeapProfiling()`**(실험적) 은 기록에 메모리 데이터를 더해 줍니다.

블랙박스라고 생각하면 돼요. 조사할 때만 켜고, 기록은 짧고 집중되게 유지하며, 진짜 가치는 녹화 자체가 아니라 나중에 그 테이프를 읽는 데 있어요. 이름 두 개 — **`startRecording`** 과 **`stopRecording`** — 만 기억하면 90% 는 끝난 셈이에요.

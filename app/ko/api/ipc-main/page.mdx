`ipcMain` 모듈은 **Main 프로세스** 에 살아요. 렌더러 프로세스(여러분의 웹 페이지)가 이름 붙은 **채널** 로 보내는 메시지를 받는 `EventEmitter` 이고, Main 프로세스가 다시 답할 수 있게 해 줘요. 공유되는 단 하나의 객체라서 `new` 로 만들지 않아요.

```js filename="main.js"
const { ipcMain } = require('electron')

// 창이 보낸 "보내고 끝" 메시지를 듣기
ipcMain.on('save-file', (event, contents) => {
  writeToDisk(contents)
})

// 요청에 답하고 창으로 값을 돌려주기
ipcMain.handle('read-settings', async (event) => {
  return await loadSettings()
})
```

<EasyNote>
`ipcMain` 은 Main 프로세스의 **프런트 데스크** 예요. 렌더러 창(손님)이 특정 채널에서 벨을 누르면, `ipcMain` 이 그 벨을 듣고 무엇을 할지 정하는 안내원이에요. 어떤 때는 그냥 메모만 받고(`on`), 어떤 때는 답을 건네줘요(`handle`).
</EasyNote>

## 두 가지 패턴

창이 `ipcMain` 과 대화하는 방법은 두 가지이고, 알맞은 쪽을 고르는 게 핵심이에요:

| 패턴 | 렌더러가 호출 | Main 의 응답 | 언제 쓰나 |
|---|---|---|---|
| **단방향** | `ipcRenderer.send(채널, ...)` | `ipcMain.on(채널, 리스너)` | Main 에 _알리기만_ 할 때 (로그, "이거 저장해", "X 눌렀어") |
| **요청/응답** | `ipcRenderer.invoke(채널, ...)` | `ipcMain.handle(채널, 리스너)` | _값을 돌려받아야_ 할 때 (파일 읽기, DB 조회, 대화상자 결과) |

<Callout type="info">
`ipcMain` 에는 독립적인 이벤트나 속성이 없어요 — 전체 표면이 아래의 메서드들이에요. 여기서 다루는 "이벤트"는 `on`/`handle` 을 연결할 때 **여러분이 직접 짓는 채널 이름** 이에요.
</Callout>

## 메서드(Methods)

### 듣기 (단방향 메시지)

`ipcRenderer.send` 로 보낸 메시지를 위한 핸들러를 연결해요. 리스너는 `리스너(event, ...args)` 형태로 호출돼요.

| 메서드 | 하는 일 |
|---|---|
| `ipcMain.on(channel, listener)` | `channel` 을 듣고, **모든** 메시지마다 리스너 실행 |
| `ipcMain.once(channel, listener)` | `on` 과 같지만, **한 번** 실행 후 리스너 제거 |
| `ipcMain.addListener(channel, listener)` | `ipcMain.on` 의 별칭 |
| `ipcMain.off(channel, listener)` | `channel` 에서 특정 `listener` 제거 |
| `ipcMain.removeListener(channel, listener)` | `ipcMain.off` 의 별칭 |
| `ipcMain.removeAllListeners([channel])` | `channel` 의 모든 리스너 제거 (생략하면 **모든** 채널) |

```js filename="listen.js"
const { ipcMain } = require('electron')

// 단방향: 렌더러가 알려 주고, 반환값으로 답하지 않아요
ipcMain.on('counter:increment', (event, amount) => {
  // event.sender 는 메시지를 보낸 WebContents 예요
  total += amount
  // 원하면 같은 창으로 갱신을 밀어 보낼 수 있어요
  event.sender.send('counter:updated', total)
})

// 자동 정리: 맨 처음 한 번만 관심 있을 때
ipcMain.once('app:first-paint', (event) => {
  markStartupComplete()
})
```

<EasyNote>
리스너의 첫 번째 인자는 항상 `event` 예요 — 여러분의 데이터가 **아니에요**. 데이터는 그 뒤에 와요. 초보자가 흔히 `(amount) => ...` 라고 쓰고 `amount` 가 이상한 객체라며 헤매는데, 그 객체가 바로 `event` 예요.
</EasyNote>

### 요청 / 응답 (invoke 핸들러)

`ipcRenderer.invoke` 호출에 답해요. 리스너가 반환하는 값(값 또는 `Promise`)이 렌더러의 `invoke` 프로미스가 resolve 되는 값으로 돌아가요.

| 메서드 | 하는 일 |
|---|---|
| `ipcMain.handle(channel, listener)` | `ipcRenderer.invoke(channel)` 에 **값을 반환** 하는 핸들러 등록 |
| `ipcMain.handleOnce(channel, listener)` | `handle` 과 같지만, **한 번** 호출 후 핸들러 제거 |
| `ipcMain.removeHandler(channel)` | `channel` 에 등록된 핸들러 제거 |

```js filename="handle.js"
const { ipcMain, dialog } = require('electron')

// 요청/응답: 렌더러가 돌아올 값을 await 해요
ipcMain.handle('dialog:openFile', async (event) => {
  const { canceled, filePaths } = await dialog.showOpenDialog({
    properties: ['openFile'],
  })
  if (canceled) return null
  return filePaths[0] // 렌더러의 `await ipcRenderer.invoke('dialog:openFile')` 를 resolve
})

// 일회성 핸드셰이크 — 첫 invoke 후 자동 제거
ipcMain.handleOnce('app:get-token', () => generateOneTimeToken())
```

<Callout type="warning">
`handle`/`handleOnce` 는 채널당 **하나** 의 핸들러만 등록할 수 있어요. 같은 채널에 `handle` 을 또 호출하면 예외가 나요 — 다시 등록하려면 먼저 `removeHandler(channel)` 을 호출하세요.
</Callout>

<Callout type="warning">
`handle` 리스너가 예외를 던지면 렌더러에는 에러의 **메시지** 만 전달돼요 — 스택 트레이스와 커스텀 속성은 직렬화 과정에서 사라져요. 핸들러를 try/catch 로 감싸고, 렌더러가 상세 정보를 필요로 하면 구조화된 에러 형태로 반환하세요.
</Callout>

## event 객체

모든 리스너는 첫 번째 인자로 `event` 를 받아요. `on`/`once` 에서는 `IpcMainEvent`, `handle`/`handleOnce` 에서는 `IpcMainInvokeEvent` 예요. 가장 유용한 멤버들:

| 멤버 | 무엇인가 |
|---|---|
| `event.sender` | 메시지를 보낸 `WebContents` — `event.sender.send(채널, ...)` 로 데이터를 다시 밀어 보냄 |
| `event.senderFrame` | 메시지를 보낸 `WebFrameMain` (iframe 에서 유용) |
| `event.reply(channel, ...args)` _(IpcMainEvent 전용)_ | 메시지를 보낸 **바로 그 프레임** 에 안전하게 답함 (프레임 간에는 `sender.send` 보다 권장) |
| `event.returnValue` _(IpcMainEvent 전용)_ | **동기** `ipcRenderer.sendSync` 호출에 답하려면 여기에 설정 |
| `event.ports` | 전달된 `MessagePortMain` 객체들 |
| `event.processId` / `event.frameId` | 보낸 프로세스·프레임을 식별하는 ID |

```js filename="reply.js"
const { ipcMain } = require('electron')

// 동기 응답 (렌더러를 멈춤 — 아껴서 쓰세요!)
ipcMain.on('get-app-version', (event) => {
  event.returnValue = app.getVersion()
})

// 프레임 안전 비동기 응답: 물어본 바로 그 프레임에 답함
ipcMain.on('ping', (event) => {
  event.reply('pong', Date.now())
})
```

<Callout type="warning">
`event.sender.send()` 는 창의 **메인 프레임** 만 겨냥해요. 메시지가 `<iframe>` 이나 하위 프레임에서 왔다면, 답이 실제로 물어본 프레임에 닿도록 `event.reply()` (또는 `event.senderFrame.send()`) 를 쓰세요.
</Callout>

## 쉽게 풀어 설명

Main 프로세스를 호텔의 **백오피스** 라고 생각해 보세요. 각 창은 객실의 손님이에요. 손님이 직접 백오피스로 걸어 들어올 수는 없어요 — 그러면 아수라장(이자 보안 구멍)이 되니까요. 대신 손님은 전화를 쓰고, `ipcMain` 이 전화를 받는 교환원이에요.

전화에는 두 종류가 있어요:

1. **"그냥 알려 드리는데요..."** — 손님이 "방 불 껐어요" 하고 끊어요. 답이 필요 없죠. 이게 `ipcRenderer.send` → `ipcMain.on` 이에요. 단방향, 보내고 끝.
2. **"하나만 확인해 주실래요?"** — 손님이 "내일 날씨가 어때요?" 하고 답을 기다려요. 이게 `ipcRenderer.invoke` → `ipcMain.handle` 이에요. 교환원이 알아보고 답을 읽어 주면, 손님의 `await` 가 그 순간 끝나요.

두 가지 규칙만 지키면 사고가 안 나요. 첫째: **교환원의 메모지에는 메시지보다 통화 자체가 먼저 적혀요** — 그래서 리스너의 첫 번째 인자가 `event` 이고, 진짜 데이터는 그 뒤에 와요. 둘째: **`handle` 은 주제당 교환원 한 명만** — 같은 채널에 두 번째 핸들러를 등록하면 Electron 이 예외를 던져요. 누구의 답을 읽어 줘야 할지 알 수 없으니까요.

세 이름만 외우면 끝이에요: **`on`**(메시지 듣기), **`handle`**(요청에 답하기), **`event.sender`**(누가 전화했나). 나머지는 여러분이 직접 짓는 채널 이름일 뿐이에요.

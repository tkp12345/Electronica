## SemVer 한눈에

`MAJOR.MINOR.PATCH` — 예) `29.2.0`

| 자리 | 언제 올라가나 |
|---|---|
| **MAJOR** | 호환성을 깨는 변경 (`require → import` 강제 등) |
| **MINOR** | 호환되는 새 기능 추가 |
| **PATCH** | 버그 수정 + 보안 패치 |

<EasyNote>
"내 코드 그대로 두고 올려도 안 깨질 거야?" → MINOR/PATCH 는 OK, MAJOR 는 _공식 마이그레이션 노트 읽고 옮겨야_ 함.
</EasyNote>

## 릴리스 채널

- **Stable** — 운영 빌드는 여기서
- **Beta** — 다음 메이저의 미리보기
- **Alpha** — 매우 빠른 실험판
- **Nightly** — 매일 빌드

## 지원 기간

각 메이저 버전은 일반적으로 _현재 + 직전 2개_ (총 3개) 가 보안 패치를 받습니다. 그 외 버전은 보안 픽스가 더 이상 백포트되지 않을 수 있어요.

## 업그레이드 추천 주기

- 분기마다 (3개월) 한 번씩 최신 stable 로 점검
- 한 번에 두 메이저 이상 건너뛰지 마세요 — 마이그레이션이 누적되어 어려워짐

## 쉽게 풀어 설명

버전 번호 `29.2.0` 같은 세 자리는 우연이 아니라 _약속_ 이에요. 첫 자리(메이저) 가 바뀌면 _큰 변화_, 가운데(마이너) 는 _부드러운 새 기능_, 마지막(패치) 는 _버그 고침_. 이 약속을 SemVer 라고 하고, npm 의 거의 모든 패키지가 이 규칙을 따라요. 이 규칙 덕에 우리는 "버전 올려도 코드 안 깨질까?" 를 _숫자만 보고_ 어느 정도 예상할 수 있어요.

Electron 은 _8주마다 새 메이저_ 를 내요. 빠르죠. 모든 메이저를 따라가는 건 비현실적이라서 보통 _현재 + 직전 2개_ (총 3개 메이저) 만 보안 패치를 받아요. 즉 우리 앱이 _3개 메이저보다 더 뒤에 머물면_ 새 취약점이 발견돼도 패치가 안 와요. 분기에 한 번씩 점검하는 습관이 안전한 이유예요.

가장 자주 하는 실수는 _두 메이저 이상 한 번에 점프_ 하는 거예요. 27 → 30 처럼 건너뛰면, 그 사이의 모든 breaking change 가 _누적_ 돼 마이그레이션 지옥이 와요. 27 → 28 → 29 → 30 처럼 _한 메이저씩만_ 가세요. 한 단계당 1~2일이면 끝나지만, 3단계를 한 번에 모으면 일주일도 모자라요.

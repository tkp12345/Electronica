## 데스크톱 앱은 오프라인을 기대한다

브라우저 사용자는 "오프라인이면 안 되는 게 당연" 정도지만, 데스크톱 앱 사용자는 _"인터넷 없을 때도 노트는 보이겠지?"_ 라고 기대해요. 그 기대를 만족시키는 게 베스트 프랙티스의 시작입니다.

## 4가지 축

| 축 | 한 줄 |
|---|---|
| **로컬 저장** | SQLite, LevelDB, IndexedDB — 무엇이든 _읽을 수 있는 사본_ 을 유지 |
| **요청 큐** | 오프라인일 때 한 일을 줄 세웠다가, 온라인 되면 자동 동기화 |
| **상태 신호** | `online`/`offline` 이벤트로 UI 가 명확히 "지금 오프라인" 표시 |
| **재시도 정책** | 지수 백오프(exponential backoff) — 무한 재시도 금지 |

## 온오프 감지

```js filename="renderer.js"
window.addEventListener('online', () => syncQueue())
window.addEventListener('offline', () => showOfflineBanner())
```

<EasyNote>
브라우저의 `navigator.onLine` 은 _Wi-Fi 가 연결되었나_ 정도만 봐요. 진짜 서버가 살았는지 확인하려면 직접 ping 을 보내는 헬스체크가 필요해요.
</EasyNote>

## 큐 패턴 (간단 버전)

```js filename="renderer.js"
const queue = []

async function send(msg) {
  if (navigator.onLine) {
    await fetch('/api', { method: 'POST', body: JSON.stringify(msg) })
  } else {
    queue.push(msg)
  }
}

async function flush() {
  while (queue.length) {
    const next = queue[0]
    await fetch('/api', { method: 'POST', body: JSON.stringify(next) })
    queue.shift()
  }
}

window.addEventListener('online', flush)
```

## 캐시 무효화

- 사용자가 _읽기_ 만 할 거면 캐시를 적극 사용
- 쓰기/수정이 있다면 서버 응답으로 캐시 무효화하는 명확한 키 설계

## 쉽게 풀어 설명

비행기 안이나 지하철 환승 통로처럼 _인터넷이 갑자기 끊기는_ 상황에서도 앱이 _계속 쓸 만하게_ 동작했으면 좋겠다 — 이게 "오프라인 우선" 사고방식이에요. 웹사이트는 끊기면 _빙글빙글 로딩만_ 도는 게 자연스럽지만, 데스크톱 앱 사용자는 _"내 노트는 그대로 보여야지"_ 하고 기대해요.

이 기대를 만족시키는 데 필요한 네 가지 도구가 있어요. ① 로컬 저장(SQLite·IndexedDB·LevelDB) 으로 _읽을 수 있는 사본_ 을 유지, ② 작업 큐로 _오프라인 때 한 일을 줄 세웠다가_ 온라인 복귀 시 자동 동기화, ③ `online`/`offline` 이벤트로 UI 가 _지금 오프라인_ 임을 명확히 표시, ④ 지수 백오프(exponential backoff) 로 _재시도 간격을 점점 길게_ 가져가서 서버 과부하 방지.

주의할 점 하나: `navigator.onLine` 은 _Wi-Fi 연결됐는지_ 정도만 봐요. Wi-Fi 는 연결됐는데 서버가 죽어 있을 수도 있죠. 그래서 정말 신뢰가 필요하면 우리 서버에 직접 _작은 핑(ping)_ 을 한 번 쏴서 살아 있는지 확인하는 헬스체크를 따로 두는 게 좋아요. 짧은 코드 한 줄(`await fetch('/api/ping')`) 이면 충분.

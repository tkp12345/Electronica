## 정체

Node.js 프로그램 그 자체입니다. `package.json` 의 `main` 필드가 가리키는 파일이 시작점이에요.

```js filename="main.js"
const { app, BrowserWindow } = require('electron')

app.whenReady().then(() => {
  new BrowserWindow().loadURL('https://example.com')
})
```

## 할 수 있는 일

- 창(`BrowserWindow`)을 만들고/닫기
- 메뉴/트레이/대화상자/단축키
- Node 표준 라이브러리 전부 (`fs`, `path`, `child_process`, `net` ...)
- npm 패키지 자유롭게 (`axios`, `better-sqlite3` 등)
- IPC 채널 핸들러 등록 (`ipcMain.handle`)

## 할 수 없는 일

- DOM 을 직접 만지기 — Main 에는 `document` 가 없음
- 화면을 직접 그리기 — 그건 Renderer 의 영역

<EasyNote>
Main 안에서 `document.getElementById(...)` 같은 걸 쓰면 곧장 에러예요. _감독이 무대 위로 뛰어 올라가 연기하려는 격_.
</EasyNote>

## 라이프사이클 이벤트 (자주 쓰는 것)

| 이벤트 | 언제 |
|---|---|
| `ready` (= `app.whenReady()`) | 모든 모듈 로드 끝, 창을 만들 수 있는 시점 |
| `window-all-closed` | 마지막 창이 닫혔을 때 — 보통 여기서 `app.quit()` |
| `activate` (macOS) | dock 아이콘 클릭으로 앱이 다시 활성화 |
| `before-quit` | 종료 직전 — 저장 안 한 데이터 처리에 유용 |
| `will-quit` | 종료 직전의 마지막 단계 — 단축키 해제 등 |

## 쉽게 풀어 설명

Main 프로세스는 우리 앱이 켜질 때 _가장 먼저_ 일어나서 _가장 늦게_ 잠드는, 단 하나뿐인 일꾼이에요. 비유하자면 학교 행사의 _진행자_ 같아요. 무대를 직접 만들지 않고, 마이크를 잡지도 않지만, 누가 언제 등장할지·조명은 언제 켤지·간식은 어디로 보낼지 모두 진행자가 결정해요.

Main 의 정체는 _Node.js 프로그램_ 그 자체예요. 그래서 npm 에서 받은 어떤 패키지든 쓸 수 있고, 파일·네트워크·OS 깊은 기능에 자유롭게 접근해요. 대신 한 가지를 _못_ 합니다. 화면을 직접 그리지 못해요. `document.getElementById` 같은 걸 부르면 그대로 에러가 나요. 진행자가 무대 위로 뛰어 올라가 노래를 부르는 격이거든요.

기억해야 할 이벤트는 네 가지예요. `ready`(준비 끝, 창 만들기 시작), `window-all-closed`(마지막 창 닫힘 → 보통 종료), `activate`(맥에서 dock 다시 누름), `before-quit`/`will-quit`(종료 직전 청소). 이 네 이벤트만 알면 앱의 _수명 주기_ 가 완전히 그려져요. 다음은 무대 위 배우, [Renderer 프로세스](/ko/docs/processes/renderer).

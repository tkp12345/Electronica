## 정체

`BrowserWindow` 를 하나 만들 때마다 새 Chromium 프로세스가 따로 떠요. 안에 있는 코드는 _완벽한 웹 페이지_ 입니다 — 그냥 평범한 HTML/CSS/JS.

```html filename="index.html"
<script>
  // 여기는 Renderer — 우리가 만든 평범한 웹 페이지의 자바스크립트
  document.body.style.background = 'tomato'
</script>
```

## 갖고 있는 능력

- 화면 DOM 조작 (당연)
- 브라우저 API 모두 (`fetch`, `localStorage`, `Notification`, ...)
- preload 가 노출한 `window.api.*` 함수
- (기본 ON 인 sandbox 상황에서) Node 직접 호출 금지

<EasyNote>
Renderer 는 _크롬 탭과 똑같다_ 고 생각하세요. 우리가 일반 웹사이트 만들 때처럼 코드를 쓰면 됩니다.
</EasyNote>

## Main 과 어떻게 대화?

직접 못 합니다 — preload 를 통해서만.

```js filename="renderer.js"
// preload 가 노출해 둔 함수만 호출 가능
const reply = await window.api.ping() // → 'pong'
```

내부 흐름은 `renderer.js` → preload(window.api) → ipcRenderer → ipcMain → main.js 의 핸들러. 5단계지만 한 번 익히면 패턴이 매우 명확해요.

## 여러 Renderer 가 있을 때

각 창은 _완전히 독립된 프로세스_ 입니다. 한 창이 죽어도 다른 창은 멀쩡해요. 대신 창끼리 데이터를 공유하려면 Main 을 경유해야 합니다.

## 쉽게 풀어 설명

Renderer 는 _크롬 탭과 똑같은 공간_ 이에요. 우리가 평소에 만들던 웹페이지가 그 안에서 도는 거고, 다른 점은 _주소창이 없는 작은 창_ 이라는 것뿐이에요. 그래서 `fetch`, `localStorage`, `Notification`, `addEventListener` 같이 브라우저에서 배운 거의 모든 도구가 그대로 통해요.

대신 _컴퓨터의 깊은 곳_ — 파일 읽기·쓰기, OS 알림, 네트워크 헤더 만지기 — 은 직접 못 만져요. 그 일은 Main 의 권한이고, Renderer 는 _preload 라는 작은 창문_ 을 통해 부탁만 할 수 있어요. 코드로 보면 `window.api.save()` 같은 함수를 부르는 건데, 그 한 줄 뒤에 무전기 다섯 단계(renderer → preload → ipcRenderer → ipcMain → main 핸들러) 가 줄지어 일어나요.

마지막 포인트: BrowserWindow 를 두 개 만들면 _두 개의 Chromium 프로세스_ 가 따로 뜬다는 점. 한 창의 무한 루프가 다른 창을 멈추게 하지 않아요. 단점은 _두 창이 공유하는 변수_ 가 없다는 것. 데이터를 같이 봐야 한다면 항상 Main 을 거쳐야 해요. 다음 [Sandbox](/ko/docs/processes/sandbox) 와 [Context Isolation](/ko/docs/processes/context-isolation) 페이지가 _왜 이렇게 격리되어 있는지_ 의 이유를 설명해요.

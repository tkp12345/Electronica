## 두 종류의 프로세스

Electron 앱이 켜지면 OS 안에 _두 종류_ 의 프로세스가 생깁니다.

| 이름 | 역할 | 환경 |
|---|---|---|
| **Main** | 앱의 두뇌 — 창 생성, 메뉴, 파일 입출력, 자동업데이트 | Node.js |
| **Renderer** | 실제 화면을 그림 — HTML/CSS/JS | Chromium |

<Callout type="info" emoji="🎬">
**비유**: Main 은 _영화 감독_, Renderer 는 _배우_. 감독은 무대 뒤에서 카메라·조명을 컨트롤하고, 배우는 무대 위에서 연기를 합니다. 둘은 서로 다른 권한과 역할을 가져요.
</Callout>

## 왜 굳이 두 개?

웹 페이지에 OS 의 모든 권한을 주면 _악성 사이트가 곧장 사용자 파일을 다 읽을 수 있어요_. Chromium 이 원래 갖고 있는 "탭 = 격리된 sandbox" 개념을 그대로 가져온 거예요. **Main 만 진짜 권한을 갖고, Renderer 는 격리** 됩니다.

## 셋이 한 무대에서 만나는 그림

```text
┌────────────────────────────────────────────────────────┐
│  Main process (Node.js)                                │
│   ├─ app, BrowserWindow, ipcMain, Menu, Tray ...       │
│   └─ ipcMain.handle('save', writeFile)                 │
└──────────────────┬─────────────────────────────────────┘
                   │  IPC (channel = 'save')
                   ▼
┌────────────────────────────────────────────────────────┐
│  Renderer process (Chromium)                           │
│   ├─ 우리 웹 페이지 (HTML/CSS/JS)                       │
│   └─ window.api.save() ◀── preload 가 노출한 함수      │
└────────────────────────────────────────────────────────┘
```

## 쉽게 풀어 설명

학교 연극을 한 편 떠올려 보세요. 무대 위에서 연기하는 _배우_ 와, 무대 뒤에서 조명·음향·소품을 챙기는 _감독_ 이 있죠. 둘 다 없으면 연극이 안 돼요. 그런데 둘은 _역할이 완전히 달라요_. 배우는 관객 눈에 보이는 모든 걸 만들고, 감독은 무대 밖에서 큰 그림을 움직여요. 이 페이지의 Main 과 Renderer 도 똑같은 분업이에요.

**Main** 은 감독이에요. Node.js 로 돌아가고, 창을 몇 개 만들지·메뉴는 어떻게 그릴지·파일은 어디에 저장할지 같은 _큰 결정_ 을 담당해요. 화면을 직접 그리지는 않아요. **Renderer** 는 배우들이에요. 각 창 하나가 _독립된 Chromium_ 위에서 돌면서 실제 픽셀(HTML/CSS/JS) 을 그려요. 한 창이 다쳐도(=크래시) 다른 창은 멀쩡한 이유가 여기 있어요.

둘 사이의 대화는 **IPC**(프로세스 간 통신) 라는 _무전기_ 로만 이루어져요. 배우가 "저 무대용 칼 좀 주세요" 라고 외치면 감독이 무대 밖에서 던져 주는 거죠. 이 무전기를 잘못 다루면 보안 사고가 나기 때문에, 다음에 읽을 [Preload](/ko/docs/tutorial/preload)·[Sandbox](/ko/docs/processes/sandbox)·[Context Isolation](/ko/docs/processes/context-isolation) 페이지가 그 _안전한 무전 규칙_ 을 자세히 알려 줍니다.

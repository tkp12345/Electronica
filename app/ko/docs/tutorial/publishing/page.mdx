## 어디에 올릴까

가장 흔한 두 가지:

- **GitHub Releases** — 무료, Forge 가 기본 지원
- **자체 S3 / R2 같은 오브젝트 스토리지** — 회사 배포 채널이 별도일 때

## GitHub 으로 배포하기

<Steps>

### Forge publisher 설정

```js filename="forge.config.js"
module.exports = {
  publishers: [
    {
      name: '@electron-forge/publisher-github',
      config: {
        repository: { owner: 'your-name', name: 'your-app' },
        prerelease: false,
      },
    },
  ],
}
```

### GitHub 토큰

`GITHUB_TOKEN` 환경변수로 `repo` 권한 토큰을 넣어 둡니다.

### 한 줄로 업로드

```bash filename="terminal"
npm run publish
```

`out/` 의 모든 설치 파일이 새 Release 로 올라가요.

</Steps>

## 자동 업데이트 동작 방식

```js filename="main.js"
const { updateElectronApp } = require('update-electron-app')
updateElectronApp() // 기본값으로 GitHub Releases 확인
```

<EasyNote>
이 한 줄이 _"앱이 켜질 때마다 → 최신 버전 있는지 GitHub 에 물어보고 → 있으면 조용히 받아두고 → 사용자가 나갈 때 적용"_ 까지를 다 해 줍니다.
</EasyNote>

## 코드 서명을 꼭 같이

서명되지 않은 앱은 맥 Gatekeeper / 윈도우 SmartScreen 이 _"이 앱은 안전하지 않을 수 있어요"_ 경고를 띄워요. 진지하게 배포하려면 거의 필수입니다 (→ [Code Signing](/ko/docs/distribution/code-signing)).

## 릴리스 체크리스트

- ✅ `package.json` 의 버전 올렸나요?
- ✅ 빌드 OS 가 코드 서명 인증서를 들고 있나요?
- ✅ `update-electron-app` 이 main.js 에 한 번이라도 호출되나요?
- ✅ 릴리스 노트(CHANGELOG)에 사용자가 볼 변경점 한 줄이라도 적었나요?

## 쉽게 풀어 설명

설치 파일을 만들었다고 자동으로 사용자가 가져갈 수 있는 건 아니에요. 어딘가에 _올려 둬야_ 해요. 가장 간단한 곳이 **GitHub Releases** — 무료고, Forge 가 한 줄로 업로드해 줍니다. 회사라면 보통 자체 S3 / R2 같은 _자체 다운로드 서버_ 를 두기도 해요. 어디든 _주소_ 만 있으면 됩니다.

진짜 마법은 _자동 업데이트_ 예요. `update-electron-app` 한 줄을 main.js 에 넣어 두면, 앱이 켜질 때마다 _서버에 새 버전 있는지 물어보고, 있으면 백그라운드로 받아 두고, 사용자가 다음에 켤 때 조용히 갈아 끼우는_ 일까지 자동으로 일어나요. 사용자는 어느 날 "어, 이거 디자인 바뀌었네?" 하고 알아챌 뿐이에요.

다만 _코드 서명_ 이 없으면 자동 업데이트는 작동하지 않아요. 새 버전이 _이전 버전과 같은 서명_ 을 갖고 있어야 OS 가 "같은 앱의 새 버전이구나" 라고 받아들이거든요. 미서명 빌드는 사용자에게 위협처럼 보이고, 자동 업데이트도 멈춰요. 그래서 진지하게 배포할 거라면 [코드 서명](/ko/docs/distribution/code-signing) 페이지가 _필수 일정_ 이에요.

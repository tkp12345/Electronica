## DevTools 의 Accessibility 패널

Renderer 에서 우클릭 → "검사" 로 DevTools 를 열고, _Elements_ 탭의 우측 _Accessibility_ 섹션을 보세요. 각 노드가 스크린리더에 어떻게 들리는지 트리로 확인할 수 있습니다.

## 스크린리더로 한 번이라도 들어보기

- **macOS**: ⌘F5 → VoiceOver 켜기
- **Windows**: NVDA(무료) 가 표준
- **Linux**: Orca

<EasyNote>
"이 버튼이 _Save 버튼_ 으로 들리는지", "탭 키로 이 카드까지 도달 가능한지" — 30초만 들어봐도 빠진 레이블을 찾을 수 있어요.
</EasyNote>

## Electron 의 접근성 트리 강제 켜기

기본적으로 접근성 API 는 OS 가 _필요할 때_ 만 활성화돼요. CI 에서 강제로 켜고 싶다면:

```js filename="main.js"
app.setAccessibilitySupportEnabled(true)
```

## 쉽게 풀어 설명

만든 페이지가 _스크린리더에게 어떻게 들리는지_ 는 우리가 직접 들어 보지 않으면 모르겠죠. 다행히 두 가지 도구가 있어요. ① DevTools 의 **Accessibility 패널** — 페이지 안 모든 요소가 _어떤 역할(role) 로 들리는지_ 트리로 보여 줍니다. ② **진짜 스크린리더** — macOS 의 VoiceOver(⌘F5), 윈도우의 NVDA, 리눅스의 Orca. 30초만 켜 봐도 빠진 레이블이 한 번에 보여요.

CI 에서 자동화된 접근성 테스트를 돌릴 땐 작은 함정이 있어요. OS 는 _스크린리더가 실제로 켜져 있을 때_ 만 접근성 트리를 만들기 때문에, 자동 환경에서는 트리가 _존재하지 않는_ 상태가 돼요. `app.setAccessibilitySupportEnabled(true)` 한 줄을 main.js 에 넣으면 _강제로_ 트리가 생겨서, Playwright 같은 도구로 접근성 노드를 검사할 수 있어요.

체크리스트 한 줄 결론: _주 1회 VoiceOver 켜고 우리 앱을 5분간 써 보기_. 사용자 만족도가 가장 빨리 올라가는 습관이에요.

## 문제 — Linux 러너엔 디스플레이가 없다

`electron .` 가 _창을 띄울 곳_ 을 못 찾아 에러. Xvfb(가상 X 서버)로 _가짜 디스플레이_ 를 만들어 해결.

## GitHub Actions 예시

```yaml filename=".github/workflows/test.yml"
jobs:
  e2e:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }
      - run: npm ci
      - run: xvfb-run --auto-servernum npm test
```

<EasyNote>
`xvfb-run` 가 _보이지 않는 가상 모니터_ 를 만들어 주고, 그 위에 Electron 이 창을 띄워요. 사람 눈엔 안 보이지만 픽셀은 그려져 있어요.
</EasyNote>

## macOS / Windows 러너

이 두 OS 는 디스플레이가 기본 제공되므로 별도 설정 없이 동작합니다. 단 _GPU 가속 부재_ 로 일부 그래픽 테스트가 느리거나 다를 수 있어요.

## 트레이스 / 스크린샷 자동 수집

Playwright 의 `--trace=on` 옵션으로 _실패 시점의 동영상·HTML 트레이스_ 자동 보관 → 디버깅 절반 끝.

## 쉽게 풀어 설명

CI(자동 빌드 서버) 는 _모니터가 없는_ 작은 컴퓨터예요. 우리 개발 노트북과 똑같이 명령은 받지만, _화면 자체가 없어서_ "창을 띄워라" 같은 명령이 막혀요. 그런데 Electron 의 모든 테스트는 _창을 띄우는 것_ 부터 시작하니까, 그대로 두면 CI 에서 모든 E2E 테스트가 통째로 실패해요.

리눅스 러너의 해결책이 **Xvfb** 라는 _가짜 모니터_ 예요. 실제로 픽셀을 그리지만 _아무도 안 보는 가상 화면_ 에 그려요. `xvfb-run --auto-servernum npm test` 한 줄이면 우리 테스트가 _진짜 모니터가 있는 것처럼_ 동작해요. macOS·Windows 러너는 디스플레이가 기본 제공되니 이런 추가 설정이 필요 없어요. 단 _GPU 가속이 없어서_ WebGL 같은 그래픽 테스트는 느리거나 약간 다르게 보일 수 있어요.

마지막 꿀팁: Playwright 의 `--trace=on` 옵션을 켜 두세요. 테스트가 실패한 _바로 그 순간의 비디오·DOM 스냅샷·네트워크 로그_ 가 모두 자동 저장돼요. "왜 CI 에서만 실패하지?" 의 답을 _재현 없이_ 알 수 있어, 디버깅 시간이 절반으로 줄어요.

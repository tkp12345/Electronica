`BaseWindow` is a **class** you create with `new` in the **Main process**. It's a real OS window, but unlike `BrowserWindow` it has **no web page of its own** — instead you attach one or more `View`s (such as `WebContentsView`) to its `contentView`. This is the building block for windows that compose several pages side by side.

```js filename="main.js"
const { BaseWindow, WebContentsView } = require('electron')

const win = new BaseWindow({ width: 800, height: 600 })

// A BaseWindow holds views, not a single page
const view = new WebContentsView()
win.contentView.addChildView(view)
view.setBounds({ x: 0, y: 0, width: 800, height: 600 })
view.webContents.loadURL('https://electronjs.org')
```

<EasyNote>
Think of `BaseWindow` as an **empty picture frame with a corkboard inside it**. The frame controls size, position, and window state; the corkboard (`contentView`) is where you pin one or more pictures (`View`s). `BrowserWindow` is just a `BaseWindow` that comes with a single picture already pinned for you.
</EasyNote>

## Instance events

Listen with `win.on('event-name', handler)`. Events marked _(macOS)_ / _(Windows)_ / _(Linux)_ only fire on that platform.

### Lifecycle & visibility

| Event | Fires when |
|---|---|
| `close` | The window is about to close — `preventDefault()` to cancel |
| `closed` | The window has been closed — drop your reference, it's gone |
| `show` | The window was shown |
| `hide` | The window was hidden |
| `focus` | The window gained focus |
| `blur` | The window lost focus |
| `query-session-end` _(Windows)_ | The OS session is about to end — `preventDefault()` to delay |
| `session-end` _(Windows)_ | The OS session is ending (logout/shutdown) — cannot be prevented |

```js filename="confirm-close.js"
win.on('close', (event) => {
  if (hasUnsavedWork()) {
    event.preventDefault() // keep the window open
    showSaveDialog()
  }
})

win.on('closed', () => {
  // the window is gone — release the reference so it can be GC'd
  win = null
})
```

<Callout type="warning">
After `closed` fires, the window is destroyed — calling any method on it will throw. Always null out your reference inside the `closed` handler.
</Callout>

### Size, position & state

| Event | Fires when |
|---|---|
| `maximize` | The window was maximized |
| `unmaximize` | The window left the maximized state |
| `minimize` | The window was minimized |
| `restore` | The window was restored from minimized |
| `will-resize` _(macOS/Windows)_ | Before a resize — `preventDefault()` to block it |
| `resize` | After the window has resized |
| `resized` _(macOS/Windows)_ | A resize gesture finished |
| `will-move` _(macOS/Windows)_ | Before a move — `preventDefault()` to block it |
| `move` | The window is being moved |
| `moved` _(macOS/Windows)_ | A move gesture finished |
| `always-on-top-changed` | The always-on-top flag toggled |

### Fullscreen

| Event | Fires when |
|---|---|
| `enter-full-screen` | Entered fullscreen |
| `leave-full-screen` | Left fullscreen |

### Gestures & platform extras

| Event | Fires when |
|---|---|
| `app-command` _(Windows/Linux)_ | An App Command was invoked (e.g. media keys, browser back) |
| `swipe` _(macOS)_ | A 3-finger trackpad swipe |
| `rotate-gesture` _(macOS)_ | A trackpad rotation gesture |
| `sheet-begin` _(macOS)_ | A window sheet opened |
| `sheet-end` _(macOS)_ | A window sheet closed |
| `new-window-for-tab` _(macOS)_ | The native "new tab" button was clicked |
| `system-context-menu` _(Windows/Linux)_ | The system context menu was triggered — `preventDefault()` to suppress |

<EasyNote>
Notice what's **missing** compared to `BrowserWindow`: no `ready-to-show`, no `page-title-updated`, no `responsive` / `unresponsive`. Those are about a _web page_, and a `BaseWindow` has no page of its own. If you need them, listen on a child view's `webContents` instead.
</EasyNote>

## Instance methods

### Lifecycle

| Method | What it does |
|---|---|
| `win.destroy()` | Force-close the window **without** firing `close` |
| `win.close()` | Close gracefully (fires `close`, cancellable) |
| `win.focus()` / `win.blur()` | Focus / unfocus the window |
| `win.isFocused()` | Is this window focused? |
| `win.isDestroyed()` | `true` if the window has been destroyed |

### Content views

| Method | What it does |
|---|---|
| `win.setContentView(view)` | Set the root `View` filling the window's content area |
| `win.getContentView()` | Get the root content `View` |
| `win.setAspectRatio(ratio[, extraSize])` | Lock the resize aspect ratio (`extraSize` _(macOS)_) |

```js filename="multi-view.js"
const { BaseWindow, WebContentsView } = require('electron')

const win = new BaseWindow({ width: 1000, height: 600 })

// Compose two pages side by side inside one window
const left = new WebContentsView()
const right = new WebContentsView()
win.contentView.addChildView(left)
win.contentView.addChildView(right)

left.setBounds({ x: 0, y: 0, width: 500, height: 600 })
right.setBounds({ x: 500, y: 0, width: 500, height: 600 })

left.webContents.loadURL('https://electronjs.org')
right.webContents.loadFile('sidebar.html')
```

<Callout type="warning">
`BaseWindow` has **no** `loadURL` / `loadFile` and **no** `webContents`. You can't load a page directly into it — you must attach a `WebContentsView` and load into _that_ view's `webContents`. If you just want one page in a window, reach for `BrowserWindow` instead.
</Callout>

## In plain words

`BaseWindow` is the **window without the web page**. Where `BrowserWindow` is "a window with one browser inside it," `BaseWindow` is "a window with an empty content area you fill yourself." That one difference explains everything on this page.

1. **The frame is the same.** Size, position, maximize/minimize, fullscreen, always-on-top, the title bar, the taskbar — all of that works exactly like `BrowserWindow`, because `BrowserWindow` actually _extends_ `BaseWindow`.
2. **The inside is different.** There's no `loadURL`, no `webContents`. Instead you have `contentView`, and you attach `View`s to it — most often a `WebContentsView` (one page) or several of them laid out side by side.

So when do you reach for it? When you need **more than one web page in a single window** — a sidebar plus a main panel, a browser-style multi-tab layout, an overlay on top of content. For everything else, `BrowserWindow` is the friendlier one-page shortcut.

Remember the trade: `BaseWindow` gives you composition (**`contentView` + `addChildView`**) at the cost of doing your own page wiring; `BrowserWindow` gives you a page for free at the cost of only having one.

## Desktop users expect offline to work

Browser users tolerate "offline = broken." Desktop users assume _"my notes will still open even on a plane, right?"_ Meeting that expectation is where best practices begin.

## Four pillars

| Pillar | One line |
|---|---|
| **Local storage** | SQLite, LevelDB, IndexedDB — keep a _readable copy_ no matter what |
| **Request queue** | Queue work while offline, drain it automatically when back online |
| **Status signal** | Use `online`/`offline` events to clearly tell the user "you're offline now" |
| **Retry policy** | Exponential backoff — never an infinite retry loop |

## Detecting online/offline

```js filename="renderer.js"
window.addEventListener('online', () => syncQueue())
window.addEventListener('offline', () => showOfflineBanner())
```

`navigator.onLine` only really checks _whether Wi-Fi is up_. For "is the actual server alive?" you need a real health-check ping.
</EasyNote>

## A tiny queue pattern

```js filename="renderer.js"
const queue = []

async function send(msg) {
  if (navigator.onLine) {
    await fetch('/api', { method: 'POST', body: JSON.stringify(msg) })
  } else {
    queue.push(msg)
  }
}

async function flush() {
  while (queue.length) {
    const next = queue[0]
    await fetch('/api', { method: 'POST', body: JSON.stringify(next) })
    queue.shift()
  }
}

window.addEventListener('online', flush)
```

## Cache invalidation

- Read-only flows: cache aggressively.
- Read-write flows: design clear cache-busting keys driven by server responses.

## In plain words

When the Wi-Fi drops on a plane or while changing trains, you want the app to _still be useful_ for a while. That mindset is "offline first." Web sites have a free pass to show a spinning circle when offline, but desktop users assume their notes will keep opening, their drafts will keep saving, their cached views will keep loading.

Four tools cover that expectation. ① Local storage (SQLite, IndexedDB, LevelDB) keeps a _readable copy_ on the user's disk. ② A work queue _lines up the things they did while offline_ and drains it when the network returns. ③ The `online`/`offline` events let the UI clearly say "you're offline now." ④ Exponential backoff means the app _slowly stretches the retry interval_ so a flaky network doesn't hammer your server.

One gotcha: `navigator.onLine` really only checks _whether the Wi-Fi link is up_. The Wi-Fi can be on while your server is down. For anything important, send a small _ping_ to your own backend (`await fetch('/api/ping')`) and treat the response as the source of truth instead of trusting the OS signal alone.

## The five repeat offenders

1. **Too many `require()` calls at startup** — load only what you need before first paint.
2. **One giant Renderer bundle** — tree-shake and code-split there too.
3. **Spawning extra BrowserWindows for no reason** — each one is a full Chromium instance.
4. **DevTools open in production builds** — pure RAM cost.
5. **IPC flood** — don't fire IPC on every mouse move. Throttle/debounce.

## Measure first, guess never

```js
console.time('expensive')
doExpensive()
console.timeEnd('expensive')
```

Or open DevTools → Performance and record. _Optimizing without measuring is almost always wrong._

<EasyNote>
"It's slow" usually means one of three things: _slow to start_, _scrolling stutters_, or _switching windows lags_. Pin down which before fixing.
</EasyNote>

## Cutting startup time

```js filename="main.js"
// 🚫 sync require bomb
const heavy = require('heavy-pkg')

// ✅ lazy import when actually needed
ipcMain.handle('use-heavy', async () => {
  const { default: heavy } = await import('heavy-pkg')
  return heavy.compute()
})
```

## Cutting memory

- Enable `backgroundThrottling` on BrowserWindows to throttle background tabs.
- Call `.destroy()` on windows you'll never reuse.
- Use OS-level caches for big images/video (local cache path beats live CDN URL).

## In plain words

Where does "Electron apps are heavy" come from? The Chromium runtime itself _is_ chunky — typically 50–150MB on disk. But most of the slowness users actually feel comes from _our own code_, not the engine. Require fifty npm packages at startup, ship one enormous bundle to the Renderer, fire IPC on every mouse move — any framework would crawl under that.

Five habits do most of the work. ① _Import only when needed_ (`await import()` for lazy loading). ② _Tree-shake and code-split the Renderer too_. ③ _Don't spin up extra BrowserWindows for fun_ — each is a full Chromium instance. ④ _Block DevTools in production builds_. ⑤ _Tame IPC floods_ with throttle/debounce.

The single most important rule is the inverse of "premature optimization is the root of all evil" — here it's _measure first, never guess_. Record a session in the DevTools Performance tab and start from the _fattest red bar_. Usually that one bar accounts for _80% of the perceived slowness_. Catching one big bar beats chasing a hundred small ones every time.

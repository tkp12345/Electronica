## Why preload exists

If web pages could call `require('fs')` directly, **one malicious site could read every file on your user's machine.** Electron defaults to keeping the web side and the Node side strictly separated. When they truly need to talk, you build a small **bridge** called a preload script and expose only the exact functions you want.

<EasyNote>
Think of the "guest room" (Renderer) and the "kitchen" (Main). You don't hand the guest a knife — you cut a tiny window and pass only finished plates through it. That window is your preload.
</EasyNote>

## Add a preload

<Steps>

### Write `preload.js`

```js filename="preload.js"
const { contextBridge, ipcRenderer } = require('electron')

// Only window.api.ping() is reachable from the web page.
contextBridge.exposeInMainWorld('api', {
  ping: () => ipcRenderer.invoke('ping'),
})
```

<EasyNote>
`contextBridge.exposeInMainWorld` means _"push just these functions through the little window."_ Everything else stays in the kitchen.
</EasyNote>

### Register the preload + a handler in `main.js`

```js filename="main.js"
const { app, BrowserWindow, ipcMain } = require('electron')
const path = require('node:path')

app.whenReady().then(() => {
  // Main side: "answer 'pong' to any 'ping'."
  ipcMain.handle('ping', () => 'pong')

  const win = new BrowserWindow({
    width: 800,
    height: 600,
    webPreferences: {
      preload: path.join(__dirname, 'preload.js'),
    },
  })
  win.loadFile('index.html')
})
```

### Use it from `index.html`

```html filename="index.html"
<button id="b">Ping</button>
<pre id="out"></pre>
<script>
  document.getElementById('b').addEventListener('click', async () => {
    const reply = await window.api.ping()
    document.getElementById('out').textContent = reply
  })
</script>
```

</Steps>

## Three rules to remember

1. **Context isolation (default ON)** — keeps the web's JS and the preload's JS from sharing globals.
2. **Node integration (OFF)** — never let `require()` reach the renderer; this is rule #1 of Electron security.
3. **Sandbox (ON)** — minimize what the page itself can do.

<Callout type="warning">
**If an old tutorial enables `nodeIntegration: true`, ignore it.** That single flag is the #1 cause of Electron security incidents.
</Callout>

## In plain words

Imagine a _treasure chest_ in the middle of a schoolyard. If every kid could just walk up and grab whatever they wanted, you'd lose the contents in five minutes. So the grown-ups put the chest behind bars and add a tiny _mail slot_ next to it. You write down the name of the thing you want; an adult on the other side decides whether to hand it over.

The web page (Renderer) and the deeper parts of the computer (Main) work the same way. If the page could call dangerous functions like `require('fs')` directly, one malicious site could read every file you own. By default Electron keeps the two sides separated and only opens a small mail slot called the **preload script** between them.

The key tool is `contextBridge.exposeInMainWorld('api', { ... })`. That line means "expose only these functions to the page." The page can only order from the menu printed on the mail slot. Even if the page gets fully hacked, it can't call anything you didn't list. When an old tutorial tells you to set `nodeIntegration: true`, that's basically _leaving the door of the cage wide open_ — don't do it.

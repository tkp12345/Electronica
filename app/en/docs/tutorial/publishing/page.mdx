## Where to publish

The two common picks:

- **GitHub Releases** — free, supported by Forge out of the box.
- **Your own S3 / R2 / object storage** — when your company has a separate distribution channel.

## Publishing to GitHub

<Steps>

### Configure a Forge publisher

```js filename="forge.config.js"
module.exports = {
  publishers: [
    {
      name: '@electron-forge/publisher-github',
      config: {
        repository: { owner: 'your-name', name: 'your-app' },
        prerelease: false,
      },
    },
  ],
}
```

### GitHub token

Set `GITHUB_TOKEN` to a personal access token with `repo` scope.

### One command to upload

```bash filename="terminal"
npm run publish
```

Every installer under `out/` is uploaded to a new GitHub Release.

</Steps>

## How auto-update works

```js filename="main.js"
const { updateElectronApp } = require('update-electron-app')
updateElectronApp() // checks GitHub Releases by default
```

<EasyNote>
That one line means _"on every launch, ask GitHub if there's a newer version, download it quietly if so, and apply it the next time the user quits."_ Three big things, one call.
</EasyNote>

## Sign your code

Unsigned apps get an "unidentified developer" warning from macOS Gatekeeper and a SmartScreen prompt on Windows. For any serious release it's effectively required (→ [Code Signing](/en/docs/distribution/code-signing)).

## Release checklist

- ✅ Bumped the version in `package.json`?
- ✅ Build machine has the signing certificates loaded?
- ✅ `update-electron-app` is called at least once in main.js?
- ✅ One line in the release notes for the user to read?

## In plain words

Building an installer doesn't magically put it in your users' hands — you have to _host it somewhere_. The simplest place is **GitHub Releases** — free, and Forge uploads to it in one command. Companies often host on their own S3 / R2 / object storage too. Wherever it lives, what matters is a stable URL.

The real magic is _auto-update_. Drop `update-electron-app` into main.js and on every launch the app quietly _asks the server for a newer version, downloads it in the background, and swaps it in the next time the user quits_. From the user's side, one day they just open the app and notice "huh, the design changed."

There's a catch: without _code signing_, auto-update doesn't work. The new build must carry the _same signature_ as the running one for the OS to accept it as "the same app, just newer." Unsigned builds also scare your users with warnings. So [Code signing](/en/docs/distribution/code-signing) is effectively a required step in the timeline for any serious release.

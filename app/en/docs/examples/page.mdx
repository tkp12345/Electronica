## Three quickest entry points

| Resource | What it is | When to grab it |
|---|---|---|
| **electron-quick-start** | The minimal one-file boilerplate | First time learning the pattern |
| **Electron Fiddle** | Try Electron code in-browser | Sanity-check one API quickly |
| **electron-react-boilerplate** | React + TS + Webpack, batteries included | Production-flavored starter |

## Mini example — read the clipboard

```js filename="main.js"
const { app, BrowserWindow, clipboard, ipcMain } = require('electron')

ipcMain.handle('read-clipboard', () => clipboard.readText())
```

```js filename="preload.js"
const { contextBridge, ipcRenderer } = require('electron')
contextBridge.exposeInMainWorld('api', {
  read: () => ipcRenderer.invoke('read-clipboard'),
})
```

<EasyNote>
Even a "what's on the clipboard?" call goes through _ask Main → preload bridges it → page receives the result_. Almost every Electron interaction looks like this trio.
</EasyNote>

## In plain words

The fastest way to internalize a tool is to _wire up one small thing yourself_. The clipboard example on this page is exactly that. Asking "what did the user just copy?" builds the entire three-step pipeline at once: _Main talks to the real OS → preload bridges the result → the page renders it_. That triangle is how almost every Electron feature is shaped.

Jumping straight into a giant boilerplate (like electron-react-boilerplate) hides _which lines are yours_, and that's confusing while you're learning. A cleaner path is to start from the tiniest possible base — **electron-quick-start** — and add parts (React, TS, Redux) one at a time. Once the pattern lives in your hands, it's instantly recognizable inside any boilerplate.

If you just want to test something without building a project, **Electron Fiddle** is the answer. Two lines and a "run" button in the browser pop a real Electron window. It's a five-second sandbox for "what exactly does this API return?"

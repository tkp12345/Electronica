## How it differs from browser drag

Browser drag-and-drop stays inside the page. Electron can hook into the OS so a user can drag _from your window into Finder/Explorer_ and drop a real file.

## Kick it off from the Renderer

```js filename="renderer.js"
document.querySelector('.draggable').addEventListener('dragstart', (e) => {
  e.preventDefault()
  window.api.startDrag('/tmp/export.pdf')
})
```

## Preload + Main

```js filename="preload.js"
const { contextBridge, ipcRenderer } = require('electron')
contextBridge.exposeInMainWorld('api', {
  startDrag: (filePath) => ipcRenderer.send('ondragstart', filePath),
})
```

```js filename="main.js"
const { ipcMain } = require('electron')
const path = require('node:path')

ipcMain.on('ondragstart', (event, filePath) => {
  event.sender.startDrag({
    file: filePath,
    icon: path.join(__dirname, 'drag-icon.png'),
  })
})
```

<EasyNote>
`event.sender.startDrag` is the magic call that starts an _OS-level_ drag — something the browser cannot do.
</EasyNote>

## In plain words

Drag-and-drop inside a browser only goes as far as the _edges of the page_. You can drop an image into a different area of the same site, but you can't drag from the site into _Finder or Explorer_. Because Electron is a desktop app, it can cross that line — a drag that starts in your window becomes a _real OS file drag_, so dropping it on Finder actually saves a file there.

The key call is `event.sender.startDrag({ file, icon })`. The Renderer _can't call it directly_ (security), so the flow is one hop longer than usual: _renderer's dragstart → preload → ipcMain.on('ondragstart') → startDrag_. Once that shape clicks, it stays the same forever.

One gotcha: the drag _icon_ must be a **PNG** and supplied as an **absolute path**. A relative path leaves the OS unable to find the icon and the drag silently does nothing.

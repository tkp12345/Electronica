## Two events in the Renderer

```js filename="renderer.js"
window.addEventListener('online', () => setBanner('Online'))
window.addEventListener('offline', () => setBanner('Offline'))
```

## Detect from Main (even windowless)

```js filename="main.js"
const { BrowserWindow, app } = require('electron')

app.whenReady().then(() => {
  const hidden = new BrowserWindow({ show: false })
  hidden.webContents.executeJavaScript(`
    window.addEventListener('online', () => { /* notify via IPC */ })
  `)
})
```

<EasyNote>
Main itself has no `navigator`. A common pattern is to spin up a _hidden window_ that listens for the event and forwards it over IPC.
</EasyNote>

## Ping for real reachability

`navigator.onLine` only checks the Wi-Fi link. For "is the server actually alive?":

```js
async function isAlive() {
  try {
    const r = await fetch('/api/ping', { cache: 'no-store' })
    return r.ok
  } catch {
    return false
  }
}
```

## In plain words

Networks don't behave like a clean light switch — on or off. Wi-Fi can flap on and off every second when the signal is weak, and even when Wi-Fi is solid, _your server_ might be down. So you need two layers of detection.

The first layer is _what the OS tells you_. The browser's `window.addEventListener('online' / 'offline', ...)` events are the cheapest, fastest signal — perfect for the UI's first reaction. The second layer is _asking the server directly_ with a small ping. Sending a lightweight request every second or two catches the "Wi-Fi is fine but the server is dead" case the OS can't see.

Main itself has no `navigator`. The common workaround is to spin up a _hidden tiny window_ that listens to `online`/`offline` and forwards the events via IPC. It feels a bit awkward, but it's the most reliable pattern in practice.

## DevTools Accessibility panel

Right-click in the Renderer → Inspect → _Elements_ tab → _Accessibility_ pane on the right. You see a node-by-node view of how a screen reader hears the page.

## Try a screen reader at least once

- **macOS**: ⌘F5 toggles VoiceOver.
- **Windows**: NVDA (free) is the standard.
- **Linux**: Orca.

<EasyNote>
Listening for 30 seconds — "is this announced as a Save button?", "can Tab reach this card?" — finds missing labels faster than any audit.
</EasyNote>

## Force the accessibility tree on

The OS usually only turns it on when a screen reader runs. To force it in CI:

```js filename="main.js"
app.setAccessibilitySupportEnabled(true)
```

## In plain words

You can't really know how a screen reader hears your page until you listen with one. Two tools cover this. ① The **Accessibility panel** in DevTools shows every element as a tree of _roles_, so you can see what each thing is announced as. ② An **actual screen reader** — VoiceOver on macOS (⌘F5), NVDA on Windows, Orca on Linux. Half a minute is enough to surface most missing labels.

There's a quiet trap when running automated accessibility tests in CI. The OS only builds the accessibility tree when a screen reader is actually running — in headless CI, that tree is just _missing_. Add `app.setAccessibilitySupportEnabled(true)` to main.js to force it on, and tools like Playwright can then walk the accessibility nodes.

One-line takeaway: turn on VoiceOver _once a week_ and use your own app for five minutes. It's the single fastest habit for raising real-user satisfaction.

## Two processes, two tools

| Target | Tool |
|---|---|
| **Renderer** | Chrome DevTools (Cmd/Ctrl+Shift+I) |
| **Main** | Node Inspector + VS Code's "Attach to Node" |

## Renderer: regular web debugging

```js
// open DevTools automatically (dev mode only)
win.webContents.openDevTools({ mode: 'detach' })
```

## Main: attach the Node debugger

```bash filename="terminal"
electron --inspect=5858 .
```

VS Code launch config:

```json filename=".vscode/launch.json"
{
  "type": "node",
  "request": "attach",
  "name": "Electron Main",
  "port": 5858
}
```

<EasyNote>
The port number (`5858`) is _the door Main opens for the debugger_. VS Code knocks on the same door and walks in.
</EasyNote>

## Handy tricks

- `console.trace()` prints the call stack to "where did this come from?"
- `process.versions` is a one-liner check for Node/Electron/Chrome versions.
- `electron --enable-logging` dumps Electron internals to stdout.

## In plain words

Debugging feels _twice as tricky_ as other apps because there really are two kinds of programs running side by side. When you want to inspect the actor on stage (Renderer), pop open the **Chrome DevTools** you already know — F12 or Cmd/Ctrl+Shift+I. Variables, DOM, Network — exactly the same as debugging any web page.

When you want to peek backstage at the director (Main), the toolkit changes. You attach the Node debugger. Start the app with a port — `electron --inspect=5858 .` — and have VS Code's "Attach to Node" knock on the same port. Breakpoints, step over/into, live variable inspection all work. The habit of keeping _both debuggers open in parallel_ is one of the bigger productivity multipliers in Electron work.

Don't sleep on the small tricks either. `console.trace()` answers "where did this call come from?" in one line. `process.versions` prints Node, Electron, and Chromium versions at once. `electron --enable-logging` dumps Electron's normally-hidden internal logs to stdout, which is gold when a window mysteriously refuses to render.

## Electron's situation

Electron doesn't ship Chromium/Node verbatim — it carries _small patches_ to adapt them for desktop apps. Every major upgrade requires those patches to be _rebased_, which is non-trivial work.

## What it teaches us about our own apps

- When you have to patch a third-party library, reach for [`patch-package`](https://www.npmjs.com/package/patch-package).
- Keep patches **minimal in changes, maximal in comments** — the next person needs to know _why_.
- Send a PR upstream; remove the patch the moment it lands.

## patch-package recipe

```bash filename="terminal"
npm install --save-dev patch-package

# 1) Edit the file under node_modules/some-lib, then
npx patch-package some-lib

# 2) A patches/some-lib+1.2.3.patch file is created
# 3) Re-apply on every install
```

```json filename="package.json"
{
  "scripts": {
    "postinstall": "patch-package"
  }
}
```

<EasyNote>
A patch is a _temporary band-aid_. The moment upstream ships a fix, peel the band-aid off — stale patches make future upgrades miserable.
</EasyNote>

## Want to propose a patch to Electron itself?

Active patches live under `patches/` in the `electron/electron` repo. See CONTRIBUTING.md for the PR process.

## In plain words

A _band-aid_ on a scraped knee isn't a cure — it just stops the bleeding _right now_. Same idea here: a third-party library has a bug, the official fix isn't out yet, and you need _your_ app to work today. So you edit the library's file directly in `node_modules` and tell `patch-package` to remember the change. The tool then _applies the patch automatically_ every time anyone runs `npm install`.

The flow is simple. Edit the file inside `node_modules/some-lib`, then `npx patch-package some-lib`. A `.patch` file lands in `patches/` containing only your diff. Commit that file, add `patch-package` to your `postinstall` hook, and from then on every machine that runs `npm install` ends up with the same fix baked in.

_Don't let band-aids stay on forever._ The moment upstream ships the real fix, peel the patch off and upgrade. Stale patches stop applying cleanly as the library evolves, and a year later you can't even _remove_ them. Patches are temporary by design — open a PR upstream at the same time you create the patch, so the upstream fix and your own cleanup happen together.

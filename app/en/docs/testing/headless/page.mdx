## The problem — Linux runners have no display

`electron .` has nowhere to draw and errors out. Solve it with Xvfb, a _virtual X server_ that creates a fake display.

## GitHub Actions sample

```yaml filename=".github/workflows/test.yml"
jobs:
  e2e:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }
      - run: npm ci
      - run: xvfb-run --auto-servernum npm test
```

<EasyNote>
`xvfb-run` creates an _invisible virtual monitor_ for Electron to draw onto. Nobody sees it, but the pixels exist.
</EasyNote>

## macOS / Windows runners

Both come with displays out of the box — no extra setup. Heads up: no GPU acceleration on these runners can make some graphics tests slower or render differently.

## Auto-capture traces & screenshots

Use Playwright's `--trace=on` to keep _video + HTML traces from the moment of failure_. That's half the debugging gone already.

## In plain words

A CI server is basically a tiny computer _with no monitor_. It accepts commands just like your laptop, but since it has no screen, "open a window" fails. Every Electron test starts by opening a window, which is why an unmodified test suite breaks the moment it touches CI.

The solution on Linux runners is **Xvfb**, a _fake monitor_. It really does draw pixels — onto an invisible canvas nobody looks at. `xvfb-run --auto-servernum npm test` is the entire setup; from your test's point of view, a real monitor is there. macOS and Windows runners already provide a display, so no extra setup is needed — but heads up that _no GPU acceleration_ on these runners can make WebGL or canvas-heavy tests slower or render slightly differently.

The single biggest CI productivity trick is Playwright's `--trace=on`. It auto-captures the _video, DOM snapshot, and network log from the exact moment of failure_. "Why does it only fail in CI?" becomes answerable without reproducing locally, and debugging time drops by half.

## Why Playwright

- Launches _real Chromium_, exactly what users see.
- Selectors, traces, video recording all built in.
- WebdriverIO works too, but Playwright has the gentlest learning curve.

## Smallest end-to-end example

```js filename="tests/launch.spec.ts"
import { test, expect, _electron as electron } from '@playwright/test'

test('shows Hello World', async () => {
  const app = await electron.launch({ args: ['.'] })
  const win = await app.firstWindow()

  await expect(win.getByText('Hello, Electron')).toBeVisible()

  await app.close()
})
```

<EasyNote>
`electron.launch` _starts the real app_; `firstWindow()` _auto-focuses the first window_. From there it's regular Playwright.
</EasyNote>

## A good test split

- Keep **Main logic** in pure functions → unit-test with Jest/Vitest.
- Cover **UI behaviour** with Playwright E2E.
- **IPC handlers** are callable directly, so prefer fast unit tests there.

## In plain words

Automated tests are basically _a robot that taps your app the same way every time_. Every time you change a line of code, the robot replays _dozens of scenarios in seconds_ and tells you either "all good" or "this one broke." You don't have to click your way through 100 manual checks before each release, which means you can actually ship often.

The de facto Electron tool here is **Playwright**. It ships a special `_electron` mode — `electron.launch({ args: ['.'] })` is enough to _boot your real app_ and start interacting with the first window. Selectors, traces, video recording — all in one box.

The trick to keeping the suite healthy is _layering_. ① Pure functions in Main → fast unit tests (Jest/Vitest). ② IPC handlers can be invoked directly → also unit tests. ③ Reserve Playwright E2E for _real UI flows_ that genuinely need a browser. If you run everything through E2E, the suite gets so slow that nobody runs it.

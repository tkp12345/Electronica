## The five tabs you'll live in

| Tab | What for |
|---|---|
| **Elements** | DOM inspection, live CSS edits |
| **Console** | Logs, expressions, error stacks |
| **Network** | API and file-load traces |
| **Performance** | Frame and CPU profile |
| **Memory** | Leak hunting, heap snapshots |

## Open it automatically

```js filename="main.js"
if (process.env.NODE_ENV === 'development') {
  win.webContents.openDevTools({ mode: 'detach' })
}
```

<EasyNote>
`mode: 'detach'` opens DevTools in its own window so it doesn't cover a small app window.
</EasyNote>

## Bind your own shortcut

```js filename="main.js"
win.webContents.on('before-input-event', (event, input) => {
  if (input.key === 'F12') win.webContents.toggleDevTools()
})
```

## Block it in production

```js
if (process.env.NODE_ENV === 'production') {
  win.webContents.on('devtools-opened', () => win.webContents.closeDevTools())
}
```

## In plain words

The Chrome DevTools you pop with F12 work the same way inside your Electron app — the Renderer literally _is_ Chromium. Five tabs cover most of your day. **Elements** for the DOM, **Console** for logs and expressions, **Network** for traffic, **Performance** for slow stretches, **Memory** for leaks.

In development, opening DevTools _automatically_ saves a lot of clicks. `win.webContents.openDevTools({ mode: 'detach' })` pops it as a separate window so it doesn't cover a small app window. Binding your own shortcut (e.g. F12) also makes the muscle memory faster.

The non-negotiable rule is _block DevTools in production builds_. Letting users open DevTools inside a shipped app exposes your React tree, API calls, and env variables. Bad for security, weird for UX. A `NODE_ENV === 'production'` guard that _closes it the moment it opens_ takes care of it.

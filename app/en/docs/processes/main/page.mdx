## What it is

Just a Node.js program. The `main` field in `package.json` points at the file Electron runs first.

```js filename="main.js"
const { app, BrowserWindow } = require('electron')

app.whenReady().then(() => {
  new BrowserWindow().loadURL('https://example.com')
})
```

## What it can do

- Create and destroy windows (`BrowserWindow`)
- Menus, tray, dialogs, global shortcuts
- The full Node stdlib (`fs`, `path`, `child_process`, `net`, ...)
- Any npm package (`axios`, `better-sqlite3`, ...)
- Register IPC handlers (`ipcMain.handle`)

## What it _cannot_ do

- Touch the DOM directly — Main has no `document`.
- Draw pixels — that's the Renderer's job.

<EasyNote>
Calling `document.getElementById(...)` from Main throws immediately. _The director can't perform on stage._
</EasyNote>

## Lifecycle events you'll use

| Event | When |
|---|---|
| `ready` (via `app.whenReady()`) | All modules loaded, safe to create windows |
| `window-all-closed` | Last window closed — usually call `app.quit()` here |
| `activate` (macOS) | Dock icon clicked while the app is running |
| `before-quit` | Just before quit — save pending data |
| `will-quit` | Very last step — unregister shortcuts |

## In plain words

Main is the _single_ worker that wakes up first when the app launches and goes to sleep last. Picture the _stage manager_ at a school event. They don't build the stage or grab the microphone, but they decide when each act enters, when the lights come up, where the snacks go.

Under the hood, Main _is_ a Node.js program. So it can use any npm package, read and write files freely, talk to the network, and reach deep OS features. The one thing it _cannot_ do is draw the screen. Calling `document.getElementById` from Main throws immediately — that would be the stage manager hopping on stage to sing.

The four events worth memorizing: `ready` (you can start creating windows now), `window-all-closed` (last window gone — usually quit), `activate` (macOS Dock icon clicked again), and `before-quit` / `will-quit` (last chance to clean up). With those four you've got the entire app lifecycle. Next: the actors themselves — [Renderer process](/en/docs/processes/renderer).

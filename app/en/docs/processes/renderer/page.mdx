## What it is

Each `BrowserWindow` spawns its own Chromium process. The code inside is just a regular web page.

```html filename="index.html"
<script>
  // Renderer side ‚Äî vanilla browser JS
  document.body.style.background = 'tomato'
</script>
```

## What it can do

- Manipulate the DOM (of course)
- Browser APIs (`fetch`, `localStorage`, `Notification`, ...)
- Whatever `window.api.*` your preload exposed
- (With sandbox ON, which is the default) it cannot directly use Node

<EasyNote>
Treat the Renderer as _just a Chrome tab._ Write code the same way you would for any plain website.
</EasyNote>

## Talking to Main

Not directly ‚Äî only through the preload bridge.

```js filename="renderer.js"
// Only the functions the preload exposed are reachable
const reply = await window.api.ping() // ‚Üí 'pong'
```

The actual chain is `renderer.js` ‚Üí preload (`window.api`) ‚Üí ipcRenderer ‚Üí ipcMain ‚Üí handler in `main.js`. Five hops, but the pattern is rock-solid once you've seen it.

## Multiple Renderers

Each window is a fully independent process. If one window crashes, the others stay alive. The flip side: sharing state across windows must go through Main.

## In plain words

A Renderer is _just like a Chrome tab_, except the tab lives in your own little window without an address bar. The web page you'd normally write in any browser runs here, and almost every tool you know (`fetch`, `localStorage`, `Notification`, `addEventListener`) works exactly as you'd expect.

What it _cannot_ do is touch the deeper parts of the computer ‚Äî reading files, sending OS notifications, fiddling with network headers. Those powers belong to Main. The Renderer can only ask, through a small window called the preload. In code that looks like calling `window.api.save()`, but behind that single line is a five-step walkie-talkie chain (renderer ‚Üí preload ‚Üí ipcRenderer ‚Üí ipcMain ‚Üí main handler).

One last point worth burning in: opening two `BrowserWindow`s spins up _two separate Chromium processes_. An infinite loop in one window will not freeze the other. The flipside is that the two windows share no variables ‚Äî any shared state has to be coordinated through Main. The next pages, [Sandbox](/en/docs/processes/sandbox) and [Context Isolation](/en/docs/processes/context-isolation), explain _why_ this isolation is the default.

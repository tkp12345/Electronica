## Why it matters

If preload and the web page shared one JavaScript world, a hostile script on the page could read preload internals and grab IPC handles. **Context isolation** drops a glass wall between them.

<EasyNote>
"Two people in the same room share the desk. Put them in separate rooms and only the _service hatch (contextBridge)_ lets things move between."
</EasyNote>

## The safe way to expose things: `contextBridge`

```js filename="preload.js"
const { contextBridge, ipcRenderer } = require('electron')

contextBridge.exposeInMainWorld('api', {
  // expose functions, never raw objects
  save: (data) => ipcRenderer.invoke('save', data),
  onUpdate: (callback) => {
    ipcRenderer.on('update', (_, payload) => callback(payload))
  },
})
```

<Callout type="warning">
Don't expose the raw `ipcRenderer` or `fs` modules. Cherry-pick the functions you need.
</Callout>

## A common mistake

```js
// 🚫 wrong: hands over the whole ipcRenderer
contextBridge.exposeInMainWorld('ipc', ipcRenderer)
```

That effectively cancels your security. An attacker can now send any IPC message they like.

```js
// ✅ right: allow-list the channels you actually use
contextBridge.exposeInMainWorld('ipc', {
  invoke: (channel, ...args) => {
    const allow = ['save', 'load', 'export-csv']
    if (!allow.includes(channel)) throw new Error('forbidden')
    return ipcRenderer.invoke(channel, ...args)
  },
})
```

## In plain words

Imagine two roommates sharing a single room — same desk, same drawers, same sticky notes. If one of them leaves a password lying around, the other will see it sooner or later. **Context isolation** puts the two roommates in _separate rooms_ and only opens a _small service hatch_ in the wall between them. The web page's JavaScript and the preload's JavaScript can never share variables directly.

Why be that strict? Web pages mix in code from outside — ads, tracking, third-party libraries. If even one of those could touch the preload's `ipcRenderer` object, every other safety measure collapses. With the wall in place, you can confidently push _only the approved functions_ through the hatch.

Two rules cover correct usage. ① Don't expose raw objects like `ipcRenderer` or `fs` directly. ② Hand-pick the functions you actually need. Add an allow-list of IPC channel names and even a compromised page can't reach anything you didn't sign off on. The companion pages are [Sandbox](/en/docs/processes/sandbox) and the broader [Security best practices](/en/docs/best-practices/security).
